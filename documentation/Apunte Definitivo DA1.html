Idea rectora del curso

Una app móvil moderna se construye alrededor de una sola ecuación:

UI = f(estado)

La interfaz describe qué mostrar para un estado dado. Los eventos del usuario producen un nuevo estado. La UI se recompone. Punto.

01

Computación móvil

Qué hace que algo sea "móvil" y por qué es distinto del desktop.

La idea central

La computación móvil es la capacidad de usar servicios de cómputo —tareas que involucran hardware y software, y acceso a servicios digitales— sin que importe el cambio de ubicación del usuario. Tres pilares la hacen posible:

Comunicaciones móviles
Infraestructura de radio. El dispositivo se comunica con estaciones base y el cambio entre ellas es transparente para el usuario.
Hardware móvil
Miniaturización: CPU, pantalla táctil, sensores, conectividad inalámbrica (Wi-Fi, BT, NFC, GSM), batería. Reducido tamaño, bajo consumo.
Software móvil
SO + apps optimizados energéticamente para entornos de recursos limitados. Diseñados con el contexto del usuario en mente.
i Aclaración importante

"Móvil" no implica "siempre conectado". Un teléfono usado en casa con Wi-Fi, o sin red, sigue siendo móvil. Y también existen dispositivos inalámbricos fijos (alarmas GSM, telefonía fija inalámbrica). Movilidad ≠ conectividad continua.

Móviles vs portátiles · clasificación útil

No todos los dispositivos "que se mueven" son iguales. Una clasificación que aparece en clase:

MontadosSe integran parcialmente en otro sistema. Interactúan con otros dispositivos pero dependen del host.Android Auto en un vehículo

Los tres tamaños de Weiser

Mark Weiser, padre de la Computación Ubicua (la idea de "cualquier dispositivo, en cualquier lugar y formato"), planteó tres tamaños:

Tabs · ~1 cm · smartphones, wearables Pads · tabletas, laptops Boards · TVs, pizarrones digitales

Su tesis: la mejor computadora es un sirviente silencioso e invisible. La tecnología debería pasar desapercibida y permitir que el usuario se enfoque en lo que importa.

Características que diferencian apps móviles del software tradicional

Cuando Souza & De Aquino sistematizan el problema (lo retomamos en los papers), aparecen estas restricciones recurrentes:

Energía limitada
Batería con vida útil acotada. La app no puede ser un cerdo de recursos.
Pantalla chica

Input táctil/sensorial
Touch, voz, biometría, cámara, NFC. Inputs muy distintos al teclado+mouse.
Ancho de banda variable
Puede ir de máximo a cero en segundos.
Cambio de contexto
Ubicación, sensores, red, batería: todo cambia mientras la app corre.
Interrupción constante
Llamadas, notificaciones, pérdida de red, batería baja. Hay que diseñar para eso.
Performance limitado
CPU, memoria y red más restringidas que un PC.
Portabilidad
El mismo software debe correr en hardware muy heterogéneo.

Sistemas operativos móviles

Qué son, en qué se diferencian del SO de escritorio, y los principales del mercado.

Un sistema operativo móvil es el software base que gestiona el hardware y los recursos de un dispositivo móvil (smartphone, tablet, smartwatch), permite la interacción del usuario por interfaz táctil e interpreta las instrucciones de las aplicaciones.

Está diseñado con restricciones de energía, tamaño, conectividad y seguridad. Su prioridad es la usabilidad en movimiento y la gestión eficiente de recursos.

Distribución de appsCerrada vía stores oficialesAbierta o mixta
Actualizaciones del SOControladas por fabricante / operadorControladas por usuario o admin

Panorama del mercado

Android domina el mercado mundial con una participación cercana al 83%. iOS de Apple es el segundo player. Existen alternativas como HarmonyOS (Huawei, basado en microkernel distribuido, surge tras restricciones a AOSP en EE.UU.) pero ocupan nichos.

Android no se usa solo en teléfonos: también corre en tablets, Smart TVs (Android TV / Google TV), autos (Android Auto), wearables (Wear OS) y dispositivos industriales/IoT.

03

Native, Web, Hybrid

Los tres modelos de implementación de una app mobile.

Toda app móvil se construye con uno de estos tres enfoques. La elección define capacidades, costo, performance y reach.

⬢ Native

Implementadas con el SDK del fabricante (Android Studio + Kotlin/Java, Xcode + Swift). Una app por plataforma.

+ Acceso completo al hardware y APIs · Mejor performance · UI fiel a guidelines
− Una codebase por plataforma · Más caras de mantener

⬢ Web

HTML, CSS, JavaScript ejecutadas en el browser móvil. Mismo código en todas las plataformas.

+ Una sola codebase · Disponibles en cualquier plataforma con browser
− Sin acceso pleno al hardware · No funcionan offline (sin PWA) · Performance inferior

⬢ Hybrid

Combinan native + web. Un wrapper nativo que renderiza una webview o un runtime común (React Native, Flutter, Ionic).

+ Una sola codebase con acceso (limitado) al hardware
− Performance entre native y web · Dependés del framework

Profundizar · cómo decidir entre los tres

El criterio se reduce a tres preguntas:

  1. ¿Necesito hardware específico? (cámara avanzada, sensores precisos, AR, AI on-device, juegos 3D) → Native.
  2. ¿Vendo desde la marca con presupuesto chico y bajo uso de hardware? → Hybrid o Web (PWA).
  3. ¿Quiero llegar al máximo de usuarios sin instalación? → Web.

Este curso es Android nativo con Kotlin + Jetpack Compose. Es el camino que exige más rigor pero da el mejor producto.

04

Arquitectura de Android

Las capas que componen el SO. Lo más importante de toda esta sección.

Android está organizado en capas. Como desarrollador trabajás casi exclusivamente con las dos más altas (apps y framework), pero entender el stack completo te explica por qué ciertas cosas funcionan así.

📱 Aplicaciones · tus apps + las del sistema ↕ 🧰 Application Framework · clases de alto nivel (ActivityManager, ViewSystem, Telephony, ContentProviders…) ↕ Binder IPC ↕ ⚙ Android Runtime (ART) + Librerías nativas C/C++ · OpenGL, WebKit, libc, NDK ↕ 🔌 HAL · Hardware Abstraction Layer · módulos .so por componente físico ↕ 🐧 Linux Kernel · gestión de memoria, procesos, energía, drivers

Linux Kernel · la base

Android está construido sobre el kernel de Linux, pero profundamente modificado para las exigencias móviles. Administra recursos de hardware (memoria, procesos, red, energía) y contiene los drivers específicos del dispositivo.

  • Optimizado para ARM y ARM64 (mandatorio desde Android 5.0).
  • Cada fabricante adapta el kernel a su hardware.
  • Tiene soporte agresivo para gestión de batería.

HAL · Hardware Abstraction Layer

El HAL es la capa que abstrae el hardware del resto del SO. Cada componente físico (cámara, GPS, Bluetooth, vibrador) tiene un módulo HAL: una librería compartida (.so) escrita en C/C++.

Cómo fluye una llamada típica (ej: app usa la cámara):

App (Java/Kotlin) ↓ llama API del Framework CameraManager · ICameraService · CameraServer ↓ Binder IPC Camera HAL (módulo .so) ↓ Driver del kernel ↓ Hardware físico
i Por qué importa el HAL aunque no lo programes

No vas a escribir HALs como dev de apps, pero entenderlos te ayuda a comprender limitaciones de hardware, cómo funcionan dispositivos personalizados, y abre la puerta a desarrollo low-level (NDK, firmware, custom ROMs). Para apps muy críticas como Netflix o juegos pesados se usa el NDK para acceder de manera más directa al HW.

Librerías

Las librerías son conjuntos de funciones, clases o recursos reutilizables. En Android conviven dos familias:

Librerías nativas del sistema

Vienen con el SO y son fundamentales para las apps:

android.os · procesos, hilos, servicios android.view · UI android.content · recursos y contexto android.media android.bluetooth

Librerías de terceros

Las agregás vos vía Gradle en tu proyecto. Las ubicás en /lib. Por seguridad Android solo permite acceder a librerías del sistema o las integradas en tu app. Las más comunes en clase:

Retrofit / OkHttp
Comunicación HTTP con APIs REST.
Glide / Picasso
Carga eficiente de imágenes.
Room / Realm
Persistencia local estructurada.
Jetpack (AndroidX)
Conjunto moderno de componentes: arquitectura, UI, navegación, ciclo de vida, Compose.

📝 Las librerías son los bloques Lego del sistema: algunas vienen con la caja, otras las agregás vos, pero todas te permiten construir sin reinventar la rueda.

Application Framework

Capa intermedia que provee un conjunto de servicios, clases y componentes de alto nivel que vas a usar todo el tiempo desde tus apps. Las apps consumen estas APIs y, por debajo, el SO usa Binder para acceder a niveles inferiores.

Activity Manager Window Manager View System Telephony Manager Location Manager Content Providers

Binder IPC · cómo se comunican los procesos

Binder (Inter-Process Communication) es el mecanismo que permite que procesos se comuniquen de forma segura en Android. Habilita llamadas desde el Framework hacia los servicios del sistema, y de ahí a HAL y al Kernel. Los servicios gestionan ventanas, actividades, notificaciones, telefonía, etc.

05

ART · JIT · AOT

El runtime de Android: quién ejecuta tu código y cómo.

El Android Runtime proporciona el entorno que ejecuta las aplicaciones escritas en Java/Kotlin. Tu código no corre directamente sobre el hardware: pasa por un proceso de compilación y un runtime que lo interpreta o lo traduce a código nativo.

Bytecode · el formato intermedio

Un bytecode es código intermedio entre el fuente y el código máquina nativo. En Android el flujo conceptual es:

.kt / .java (fuente) ↓ kotlinc / javac .class (bytecode JVM, stack-based) ↓ D8 · convierte a formato Android .dex (Dalvik Executable, register-based, más compacto) ↓ se empaqueta en APK/AAB APK / AAB · listo para instalar

El bytecode .dex es independiente del hardware. Es el runtime del dispositivo el que lo interpreta o compila a código nativo del procesador (ARM, ARM64, x86…).

i Particularidad clave

Dalvik / ART no entienden .class, solo .dex. D8 es el puente entre el mundo JVM y el mundo Android. Sin D8/R8, no habría forma de correr código Java/Kotlin en Android.

JIT · Just-In-Time

Compila en ejecución, solo lo que se está usando. El runtime interpreta el bytecode y, cuando detecta métodos "calientes" (muy usados), los compila a código nativo y los cachea.

+ Arranque rápido
No hay que compilar todo antes de empezar. Warm-up rápido.
+ Adaptativo
Optimizaciones guiadas por el uso real. Puede re-optimizar en vivo.
+ Menos disco
Solo guarda bytecode más cache; ocupa menos que código nativo completo.
− Pausas runtime
La compilación durante la ejecución cuesta CPU y latencia inicial.
− Más RAM
Necesita memoria para la cache de código JIT.

AOT · Ahead-Of-Time

Compila antes de ejecutar, típicamente en la instalación. El bytecode se traduce a binario nativo que se guarda en el dispositivo.

+ Arranque inmediato
El código ya es nativo; sin warm-up. Performance estable.
+ Energía
Menos CPU usado en runtime → mejor batería.
+ Predecible
Latencias estables, ideal para apps críticas.
− Más disco
Hay que guardar binarios .oat / .vdex además del .dex.
− Instalación lenta
Compilar todo al instalar toma tiempo.
− Menos adaptativo
No se ajusta al uso real del usuario.

El modelo híbrido moderno · ART

El Android Runtime (ART) es el runtime por defecto desde Android 5.0. Reemplazó a Dalvik por mejor rendimiento y eficiencia energética. Lo que hace es combinar JIT + AOT en un esquema progresivo:

  1. Instalación rápida sin compilación pesada (solo verificación y "quickening" del .dex).
  2. Primera ejecución con JIT: la app arranca rápido, el runtime hace profiling y registra qué métodos son "hot".
  3. Compilación AOT en segundo plano (dex2oat) cuando el dispositivo está cargando e inactivo. Solo se compilan los métodos críticos según el perfil.
  4. Resultado: instalaciones rápidas + arranques cada vez más veloces + menor consumo de almacenamiento que un AOT total.
APK / AAB → .dex ↓ verificación / quickening Intérprete + JIT (métodos hot) → guarda perfiles ↓ idle + cargando dex2oat → archivos .oat (nativo) + .vdex (DEX verificado) ↓ CPU ejecuta código nativo

Profile-on-Cloud · el último escalón

Google adoptó una técnica que usa la nube para optimizar incluso la primera ejecución de una app:

  1. Devs suben sus apps al Play Store.
  2. Usuarios reales las ejecutan en miles de dispositivos.
  3. Google recolecta perfiles de ejecución vía telemetría: qué métodos son "hot" en la mayoría de los casos.
  4. Genera un cloud profile y lo distribuye junto con la app.
  5. Cuando vos instalás esa app, ART ya tiene la lista de métodos a compilar AOT desde el momento cero.

El resultado: arranque rápido incluso antes de que vos uses la app, sin esperar a generar un perfil local.

Profundizar · histórico Dalvik (anecdótico)

Dalvik fue el runtime original hasta Android 4.4 (2014). Era una JVM diseñada para entornos con recursos limitados. Trabajaba con .dex y usaba JIT puro: interpretaba en tiempo real y compilaba sobre la marcha las partes más usadas.

Pros: instalación rápida, menos espacio en disco. Contras: ejecución más lenta y mayor consumo de batería al estar siempre compilando. ART lo reemplazó porque escalaba mejor con apps complejas y hardware más capaz.

Build pipeline

Del código fuente al APK instalable.

Cuando hacés build de tu proyecto Android, ocurre una secuencia bien definida de pasos. Conocerla te ayuda a entender errores y a leer logs.

  1. Compilación a bytecode JVM: kotlinc y javac generan archivos .class (bytecode JVM, stack-based).
  2. Desugaring: el compilador reescribe "azúcar sintáctica" (lambdas, default methods, APIs nuevas) a una forma compatible con APIs Android antiguos. Sin desugar, tu app moderna no correría en celulares viejos.
  3. D8 / R8: convierte los .class en .dex (register-based, más eficiente para móviles).
    • D8 se usa cuando minifyEnabled es false.
    • R8 es D8 + shrinker (elimina código no usado) + obfuscator + minifier. Se usa para releases de producción.
  4. Multidex: si superás ~65K referencias de métodos, se generan múltiples classes.dex.
  5. Shrink de recursos (opcional): elimina drawables y layouts no referenciados.
  6. Firmado y empaquetado: el contenido se firma criptográficamente y se empaqueta como .apk (instalable directo) o .aab (Android App Bundle, formato preferido por Google Play).

⚠ APK vs JAR

Un JAR es un paquete Java estándar (zip de .class + manifest). Un APK es zip de .dex + recursos + AndroidManifest.xml + assets, firmado, listo para instalar en Android. No son intercambiables.

Seguridad de Android

Las capas de defensa que protegen el dispositivo y tus datos.

Android implementa seguridad en múltiples capas, no en una sola. Conocer los nombres alcanza para el examen.

Sandboxing
Cada app corre en su propio proceso aislado con UID único de Linux. No pueden leer datos de otra app salvo permiso explícito.
Permission System
Las apps deben declarar y, para los más sensibles, pedir al usuario permisos en runtime (cámara, ubicación, contactos, etc.).
SELinux
Mandatory Access Control a nivel kernel. Refuerza el sandbox: aunque una app comprometiera al sistema, SELinux limita qué puede hacer.
Verified Boot
El sistema verifica criptográficamente cada capa al arrancar. Si algo fue modificado, el boot lo detecta.
Google Play Protect
Analiza apps en tiempo real (en el dispositivo y en el store) buscando malware y comportamientos sospechosos.
Cifrado de almacenamiento
FDE (Full Disk Encryption) o FBE (File-Based Encryption). Datos cifrados en reposo.
App Signing + Verified APK
Cada APK está firmado por su desarrollador. El sistema rechaza instalaciones no firmadas o cuya firma cambió.
TLS / HTTPS
Por default Android exige tráfico HTTPS desde Android 9. Hay que configurar explícitamente para permitir HTTP plano.

Como dev, tu responsabilidad de seguridad incluye: usar HTTPS, no almacenar tokens en claro, validar entradas (incluso de fuentes "internas" como intents o deep links), pedir solo los permisos necesarios, y usar Keystore para datos sensibles.

UI · UX · CX

Diseño antes que código: los principios que separan una app usable de una frustrante.

Idea rectora

El diseño no es opcional: es el plano de la solución. Antes de escribir código, hay que pensar cómo hacerlo de manera eficiente. Eso es ingeniería. Sin diseño no hay ingeniería, solo improvisación.

Las tres capas: UI, UX, CX

UI · User Interface
La estética e interacción visual. Color, tipografía, layout, iconografía, accesibilidad. Pixel-perfect.
UX · User Experience
La lógica y usabilidad. Wireframes, prototipos, flujos, investigación de usuario, arquitectura de información.
CX · Customer Experience
La suma total de interacciones del cliente con la marca: marketing, ventas, postventa, soporte. UX es una parte del CX.

Como dev no diseñás directamente UI/UX, pero tu código impacta el CX: rendimiento (carga lenta = frustración), confiabilidad (un crash erosiona confianza), seguridad (vulnerabilidades destruyen la confianza al instante).

Principios de diseño · 8 reglas

  1. Usabilidad: el usuario debe poder completar tareas eficiente y efectivamente. Es el principio fundamental de UX.
  2. Consistencia: botones, menús, colores deben seguir un patrón constante. Variaciones arbitrarias confunden.
  3. Simplicidad: menos es más. Eliminar elementos innecesarios mejora la experiencia.
  4. Feedback: el usuario tiene que saber qué pasa. Confirmaciones, errores, animaciones de carga.
  5. Jerarquía visual: tamaño, color, espaciado guían la atención hacia lo importante.
  6. Accesibilidad: discapacidades visuales, auditivas, motoras o cognitivas. Etiquetas, contrastes, lectores de pantalla.
  7. Feedback del usuario: escuchar para mejorar. Identificar problemas y áreas de mejora.
  8. Testeo y prototipado: probar antes de lanzar. Iterar es más barato que rehacer.

UX Research · qué es

Proceso sistemático de estudiar usuarios para entender comportamientos, necesidades, motivaciones y contexto. Reduce suposiciones; basa decisiones en datos y empatía.

  • Reduce riesgo: valida la necesidad antes de invertir meses de desarrollo.
  • Ahorra dinero: corregir un problema en prototipo es infinitamente más barato que en producción.
  • Aumenta ROI: productos que resuelven problemas reales tienen mejor retención y lealtad.

Investigación cualitativa · "¿por qué?"

Entiende razones detrás del comportamiento.

Entrevistas de usuario Pruebas de usabilidad Observación contextual

Ciclo de diseño básico

  1. Idea/Concepto: surge la idea de la app.
  2. Prototipo de baja fidelidad: wireframes (a veces en papel) para probar usabilidad fundamental.
  3. Diseño visual: con la estructura validada, foco en estética y coherencia.
  4. Prototipo de alta fidelidad: prototipo interactivo con animaciones y transiciones para validación final.
  5. Desarrollo: con diseño aprobado, el equipo de ingeniería empieza a codificar.
Profundizar · breve historia de UX

En los años 80 se tenía una percepción ingenieril de UX: eficiencia, efectividad, seguridad, utilidad, aprendizaje, memorización.

A partir del año 2000 se produce el "auge del diseño": el foco se amplía a satisfacción, motivación, emoción, experiencias, felicidad y empatía. UX deja de ser solo "que funcione" y pasa a ser "que se sienta bien".

Jetpack Compose

El toolkit declarativo de Android. El cambio mental más importante del curso.

Jetpack Compose es el toolkit moderno de Android para construir interfaces nativas usando Kotlin. A diferencia del modelo tradicional basado en archivos XML y clases View, Compose permite describir la UI directamente con funciones de Kotlin llamadas funciones composables.

La idea central

UI = f(estado)

La pantalla que ve el usuario depende de los datos actuales del programa. Cuando cambian, Compose vuelve a ejecutar las partes necesarias de la UI. Ese proceso es recomposición.

De UI imperativa a UI declarativa

Tradicionalmente Android usaba un modelo imperativo: declarabas la vista en XML y desde Kotlin/Java buscabas elementos y los modificabas paso a paso.

// Imperativo: buscar y modificar
val texto = findViewById<TextView>(R.id.texto)
texto.text = "Resultado actualizado"

Compose es declarativo: declarás qué debe mostrarse para un estado dado.

@Composable
fun Saludo(nombre: String) {
    Text(text = "Hola, $nombre")
}

Si nombre cambia, Compose ejecuta de nuevo la función y actualiza la UI. No "buscás" un TextView ni "seteás" propiedades. Solo describís.

UI imperativa: buscar vista → modificar propiedad → sincronizar a mano UI declarativa: estado → función composable → descripción de la UI

Estructura básica de una app Compose

class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContent {
            MaterialTheme {
                PantallaPrincipal()
            }
        }
    }
}

@Composable
fun PantallaPrincipal() {
    Text(text = "Hola Jetpack Compose")
}

Tres niveles a distinguir:

  • Activity: sigue siendo parte del ciclo de vida clásico de Android.
  • setContent { }: el bloque que conecta la Activity con Compose.
  • Funciones @Composable: definen la interfaz.

Anatomía de un composable

Una función composable es una función Kotlin con la anotación @Composable. Esa anotación le dice al compilador que la función participa en la construcción de UI.

Tiene cinco elementos típicos:

  1. Parámetros de entrada: los datos que necesita para describir la UI. Recibirlos por parámetro la hace reutilizable.
  2. Elementos visuales: Text, Button, TextField, Image, Card… que también son composables.
  3. Layout: contenedores que organizan elementos (Column, Row, Box).
  4. Modifier: configura apariencia, tamaño, posición, interacción.
  5. Estado: datos que pueden cambiar durante la ejecución.

Layouts esenciales · Column, Row, Box

// Column: vertical
Column {
    Text("Uno"); Text("Dos"); Text("Tres")
}

// Row: horizontal
Row {
    Text("Nombre:"); Text("Ana")
}

// Box: superpone o posiciona dentro de un contenedor
Box { Text("Contenido") }

Compose transforma el estado en UI en tres etapas: composición de elementos → layout → dibujo. Hay layouts listos para usar, y se pueden construir personalizados.

Modifier · el "navaja suiza" de Compose

Es uno de los conceptos más importantes. Modifier sirve para modificar apariencia, tamaño, posición, interacción y comportamiento de un componente.

Text(
    text = "Hola",
    modifier = Modifier
        .padding(16.dp)
        .fillMaxWidth()
)

Algunos modifiers de uso constante:

.padding(16.dp) .fillMaxSize() .fillMaxWidth() .height(80.dp) .background(Color.X) .clickable {}

Patrón muy importante: pasar modifier como parámetro al composable. Permite que el componente sea reutilizable y que quien lo usa decida cómo posicionarlo.

@Composable
fun TarjetaUsuario(
    nombre: String,
    modifier: Modifier = Modifier
) {
    Card(modifier = modifier) {
        Text(text = nombre)
    }
}

// Uso desde afuera
TarjetaUsuario(nombre = "Carlos", modifier = Modifier.fillMaxWidth())
⚠ Buena práctica

Un composable no debería depender de variables globales o datos ocultos. Lo ideal es que reciba todo lo que necesita por parámetro. Si crece demasiado y maneja estado, dibuja UI y responde a eventos en una sola función, hay que descomponerlo.

Estado y recomposición

Cómo Compose detecta cambios y vuelve a dibujar la UI.

remember + mutableStateOf

El estado son datos que pueden cambiar mientras la app corre. Para que Compose los observe necesitás dos primitivas:

@Composable
fun FormularioNombre() {
    var nombre by remember { mutableStateOf("") }

    TextField(
        value = nombre,
        onValueChange = { nuevoValor -> nombre = nuevoValor }
    )
    Text(text = "Hola, $nombre")
}
  • mutableStateOf(x): crea un estado observable por Compose. Cuando lo modificás, Compose se entera.
  • remember { }: conserva el valor durante recomposiciones. Sin remember, cada vez que se ejecuta la función se reiniciaría el estado.
  • by: delegación de propiedad para escribir nombre en lugar de nombre.value.

Recomposición · el corazón del modelo

La recomposición es el proceso por el cual Compose vuelve a ejecutar funciones composables cuando cambian sus datos de entrada o el estado que observan.

@Composable
fun Contador() {
    var contador by remember { mutableStateOf(0) }

    Column {
        Text(text = "Valor: $contador")
        Button(onClick = { contador++ }) { Text("Incrementar") }
    }
}

Cada click incrementa contador. Compose detecta el cambio y recompone solo la parte que lee ese estado.

⚠ Implicancia clave

Como un composable puede ejecutarse muchas veces durante la vida de una pantalla, su cuerpo debe ser puro y predecible: recibe estado, describe UI, delega eventos. No debería contener lógica pesada, llamadas a APIs ni acceso directo a fuentes de datos.

State hoisting · "elevar" el estado

State hoisting consiste en sacar el estado de un composable hijo y pasarlo desde su padre. En lugar de que el hijo tenga estado propio, recibe un valor y un callback de cambio.

// Stateless: recibe valor y notifica cambios
@Composable
fun CampoNombre(
    nombre: String,
    onNombreChange: (String) -> Unit
) {
    TextField(value = nombre, onValueChange = onNombreChange)
}

// El padre administra el estado
@Composable
fun Formulario() {
    var nombre by remember { mutableStateOf("") }
    CampoNombre(nombre = nombre, onNombreChange = { nombre = it })
}

El patrón reemplaza una variable interna por dos parámetros:

  • value: T — el valor actual
  • onValueChange: (T) -> Unit — la función para modificarlo
Por qué importa

El composable no necesita saber si el dato viene de memoria, ViewModel, base local o backend. Solo lo muestra y notifica cambios. Cuanto más reutilizable debe ser, más conviene que sea stateless.

Stateful vs Stateless

Stateful
Guarda y administra estado internamente. Práctico para casos puntuales pero acoplado.
Stateless
Recibe estado desde afuera y emite eventos hacia afuera. Reutilizable y testeable.

Unidirectional Data Flow (UDF)

El flujo unidireccional de datos es uno de los patrones centrales de Compose:

State → UI → Event → State holder → State El estado baja hacia la UI · los eventos suben hacia quien modifica el estado

La UI no modifica el estado de negocio directamente. Emite eventos. El responsable del estado los procesa y produce un nuevo estado. Resultado: menos caminos posibles, más predictibilidad, debugging más fácil.

Patrón Screen — Content

Un patrón muy útil: separar el composable de pantalla (que conecta con la arquitectura) del composable de contenido (que solo renderiza datos y expone eventos).

@Composable
fun LoginScreen(viewModel: LoginViewModel) {
    val estado = viewModel.estado
    LoginContent(
        estado = estado,
        onUsuarioChange = viewModel::onUsuarioChange,
        onPasswordChange = viewModel::onPasswordChange,
        onIngresarClick = viewModel::ingresar
    )
}

@Composable
fun LoginContent(
    estado: LoginUiState,
    onUsuarioChange: (String) -> Unit,
    onPasswordChange: (String) -> Unit,
    onIngresarClick: () -> Unit
) {
    // solo renderiza, no conoce ViewModel
}

Ventaja: LoginContent es testeable y previewable sin necesidad de un ViewModel real.

UI State Pattern · agrupar el estado en una clase

En vez de tener decenas de mutableStateOf sueltos, conviene agruparlos en una data class que representa el estado completo de la pantalla. Manejar estados de carga / éxito / error con una sealed interface evita combinaciones inválidas:

sealed interface UsuariosUiState {
    object Loading : UsuariosUiState
    data class Success(val usuarios: List<Usuario>) : UsuariosUiState
    data class Error(val mensaje: String) : UsuariosUiState
}

Esto evita estados inválidos como tener simultáneamente "cargando = true" + "datos cargados" + "error activo".

Scaffold y Material Design

La estructura visual estándar de una pantalla Android moderna.

Material Design

Material Design es el sistema de diseño de Google. No es una colección de "botones lindos": es un lenguaje completo con reglas de color, tipografía, espaciado, formas, movimiento, jerarquía visual e interacción.

Compose ofrece una implementación nativa de Material en composables como:

Button Card Scaffold TopAppBar FloatingActionButton TextField NavigationBar Snackbar

MaterialTheme · el contenedor de estilos

Envolver toda la app en MaterialTheme { } aplica un tema visual coherente. Cada composable Material toma colores, tipografía y formas del tema activo.

setContent {
    MaterialTheme {
        PantallaPrincipal()
    }
}

Un Button dentro del tema toma colores y formas automáticamente; no hace falta especificar todo a mano.

Scaffold · estructura base de una pantalla

Scaffold es un composable que organiza los elementos comunes de una pantalla Material: barra superior, contenido principal, FAB, snackbar, barra inferior. Maneja correctamente los espacios ocupados por barras mediante paddingValues.

@OptIn(ExperimentalMaterial3Api::class)
@Composable
fun PantallaConScaffold() {

Ejemplo integrador · formulario completo

fun PantallaPrincipal() {

ViewModel

El "state holder" que mantiene la UI viva entre recomposiciones y rotaciones.

El ViewModel es el state holder más habitual para pantallas Android. Su función no es dibujar UI, sino conservar y producir estado de UI, responder eventos y coordinar la interacción con la capa de dominio o datos.

Responsabilidades del ViewModel

  • Mantener estado de UI a través de recomposiciones y cambios de configuración (rotación de pantalla, etc.).
  • Procesar eventos de la pantalla.
  • Lanzar operaciones asincrónicas con viewModelScope.
  • Invocar casos de uso o repositorios.
  • Exponer estado observable a Compose.
  • Evitar que la UI conozca detalles de persistencia, red o reglas de negocio.
class LoginViewModel(
    private val repository: UsuarioRepository
) : ViewModel() {

    var estado by mutableStateOf(LoginUiState())
        private set

    fun onUsuarioChange(usuario: String) {
        estado = estado.copy(usuario = usuario)
    }

    fun ingresar() {
        viewModelScope.launch {
            // validar y delegar en repository o use case
        }
    }
}
⚠ Límite importante

El ViewModel no debería contener lógica de infraestructura, SQL crudo, llamadas HTTP directas ni manipulación de archivos. Eso pertenece a la capa de datos.

ViewModel con StateFlow

En apps modernas se expone estado desde el ViewModel con StateFlow: un flujo observable con un valor actual. Encaja perfecto con Compose porque la UI puede observar y recomponerse al cambiar.

class UsuariosViewModel(
    private val repository: UsuarioRepository
) : ViewModel() {
    private val _estado = MutableStateFlow<UsuariosUiState>(UsuariosUiState.Loading)
    val estado: StateFlow<UsuariosUiState> = _estado

    init { cargarUsuarios() }

    private fun cargarUsuarios() {
        viewModelScope.launch {
            try {
                val usuarios = repository.obtenerUsuarios()
                _estado.value = UsuariosUiState.Success(usuarios)
            } catch (e: Exception) {
                _estado.value = UsuariosUiState.Error("No se pudo cargar")
            }
        }
    }
}

// En Compose:
@Composable
fun UsuariosScreen(viewModel: UsuariosViewModel) {
    val estado by viewModel.estado.collectAsState()
    UsuariosContent(estado = estado)
}

Eventos vs estado

Una pantalla no solo muestra estado: recibe acciones. Una estrategia robusta es modelar acciones como eventos (sealed interface) y manejar separadamente los one-shot events (navegar, mostrar snackbar) usando un SharedFlow que se consume con LaunchedEffect.

⚠ Error frecuente

Guardar eventos únicos como un campo del estado (ej: navegar = true). Si no se consume correctamente, puede repetirse al recomponer o al recrear la pantalla. Para eventos one-shot, usar SharedFlow.

Side Effects

Cómo correr código que no es "describir UI" desde Compose, sin romper el modelo.

En Compose, un side effect es una operación que ocurre fuera de la composición pura: iniciar una corrutina, mostrar un snackbar, registrar un listener, navegar, interactuar con APIs externas. Son necesarios pero deben usarse de forma controlada porque rompen la pureza del modelo declarativo.

LaunchedEffect

Ejecuta una corrutina asociada al ciclo de vida del composable. Si el composable sale, la corrutina se cancela. Si la key cambia, se cancela la anterior y se lanza una nueva.

@Composable
fun PantallaDetalle(id: Int, viewModel: DetalleViewModel) {
    LaunchedEffect(id) {
        viewModel.cargarDetalle(id)
    }
    DetalleContent(estado = viewModel.estado)
}

Cada vez que cambia id, se vuelve a cargar el detalle.

DisposableEffect

Para registrar y liberar recursos. Tiene un bloque onDispose que se ejecuta cuando el composable sale o cuando cambia la key.

@Composable
fun SensorScreen(sensorManager: SensorManager) {
    DisposableEffect(Unit) {
        val listener = crearListener()
        sensorManager.registerListener(listener)
        onDispose {
            sensorManager.unregisterListener(listener)
        }
    }
}

rememberCoroutineScope

Permite lanzar una corrutina en respuesta a un evento de UI (no automáticamente).

@Composable
fun PantallaConSnackbar(snackbarHostState: SnackbarHostState) {
    val scope = rememberCoroutineScope()

    Button(onClick = {
        scope.launch { snackbarHostState.showSnackbar("Guardado") }
    }) { Text("Guardar") }
}
⚠ Riesgo

Abusar de side effects rompe el modelo declarativo. Usalos solo para efectos relacionados con la UI y de forma controlada. Llamar a viewModel.cargarDatos() directamente en el body de un composable es un error: se ejecutaría en cada recomposición.

Single Source of Truth

Una parte del sistema debe ser la fuente autorizada de un dato. Múltiples copias independientes producen errores de sincronización imposibles de debuggear.

Ejemplo incorrecto: la UI tiene una lista, Room tiene otra, la API otra, el ViewModel otra.

Ejemplo ordenado: Room es la fuente local observable. La API actualiza Room. La UI observa Room.

API → Repository → Room → ViewModel → UI

Offline First

Diseñar la app para funcionar aunque la red no esté disponible. La UI lee siempre desde la base local, y el Repository sincroniza con backend cuando puede.

class UsuarioOfflineRepository(
    private val dao: UsuarioDao,
    private val api: UsuarioApi
) : UsuarioRepository {

    override fun observarUsuarios(): Flow<List<Usuario>> {
        return dao.observarUsuarios()
            .map { lista -> lista.map { it.toDomain() } }
    }

    suspend fun sincronizar() {
        val remotos = api.obtenerUsuarios()
        dao.reemplazarTodos(remotos.map { it.toEntity() })
    }
}

Ventaja: el usuario puede seguir usando la app sin conexión, y la sincronización es una responsabilidad explícita del Repository.

Repository / DataSource

Cómo aislar la UI del origen real de los datos.

Repository Pattern

El Repository abstrae el origen de los datos. La UI y el ViewModel no deberían saber si los datos vienen de Room, Retrofit, DataStore, memoria, archivos o backend.

ViewModel → Repository → DataSource 
interface UsuarioRepository {
    suspend fun obtenerUsuarios(): List<Usuario>
    suspend fun guardarUsuario(usuario: Usuario)
}

class UsuarioRepositoryImpl(
    private val localDataSource: UsuarioLocalDataSource,
    private val remoteDataSource: UsuarioRemoteDataSource
) : UsuarioRepository {

    override suspend fun obtenerUsuarios(): List<Usuario> =
        localDataSource.obtenerUsuarios()

    override suspend fun guardarUsuario(usuario: Usuario) {
        remoteDataSource.guardarUsuario(usuario)
        localDataSource.guardarUsuario(usuario)
    }
}

Esto permite cambiar Room por una API, agregar cache, implementar sincronización o crear repositorios falsos para pruebas sin tocar la UI.

DataSource Pattern

Un DataSource representa una fuente concreta de datos:

UsuarioLocalDataSource → Room UsuarioRemoteDataSource → Retrofit UsuarioPreferencesDataSource → DataStore
Diferencia clave

El DataSource sabe cómo hablar con una fuente concreta. El Repository decide cómo combinar fuentes, resolver conflictos, transformar datos y exponer una API coherente al resto de la app.

DTO, Entity, Domain Model

Es preferible no usar la misma clase para todo. En arquitectura limpia se distinguen tres modelos del mismo concepto:

DTO
Modelo usado por la API remota. Refleja el JSON/contrato del backend.
Entity
Modelo usado por Room. Refleja la tabla local.
Domain Model
Modelo usado por la lógica de la app. Limpio, sin dependencias de Room ni de la API.
data class UsuarioDto(val id: Int, val fullName: String, val email: String)

@Entity(tableName = "usuarios")
data class UsuarioEntity(@PrimaryKey val id: Int, val nombre: String, val email: String)

data class Usuario(val id: Int, val nombre: String, val email: String)

Entre ellos viven los mappers: funciones que convierten de un modelo al otro. Si el backend cambia el nombre de un campo, solo cambiás el mapper, no toda la app.

Use Case Pattern

Un Use Case encapsula una regla de aplicación o negocio. Evita que el ViewModel se transforme en una clase con demasiada lógica.

class LoginUseCase(private val repository: UsuarioRepository) {
    suspend operator fun invoke(usuario: String, password: String): Result<Usuario> {
        // validaciones, reglas de negocio, llamada al repo
    }
}

class LoginViewModel(private val loginUseCase: LoginUseCase) : ViewModel() {
    fun ingresar() {
        viewModelScope.launch {
            val result = loginUseCase(estado.usuario, estado.password)
        }
    }
}

Beneficio: las reglas se testean sin Compose ni ciclo de vida de Android.

Las capas internas no deberían depender de detalles externos. El dominio no debería conocer Compose, Room, Retrofit ni Android Context. Si tu Use Case importa androidx.compose.*, está mal acoplado.

Inyección de dependencias (Hilt)

No crees objetos: que alguien te los entregue.

La inyección de dependencias consiste en entregar a una clase los objetos que necesita, en lugar de crearlos internamente. En Android se usa frecuentemente Hilt.

Sin inyección:

class LoginViewModel : ViewModel() {
    private val repository = UsuarioRepositoryImpl()  // acoplado!
}

Problema: el ViewModel queda acoplado a una implementación concreta. No podés cambiarla por una falsa para tests.

Con inyección:

class LoginViewModel(
    private val repository: UsuarioRepository  // solo conoce la interfaz
) : ViewModel()

Ahora podés inyectar la implementación real, una falsa para tests, una con cache, etc. El ViewModel no sabe ni le importa.

Hilt en pocas anotaciones

@HiltViewModel
class LoginViewModel @Inject constructor(
    private val loginUseCase: LoginUseCase
) : ViewModel()

@Module
@InstallIn(SingletonComponent::class)
object AppModule {
    @Provides
    fun provideRepository(...): UsuarioRepository = UsuarioRepositoryImpl(...)
}

Hilt resuelve automáticamente quién provee qué cuando se construye un ViewModel.

Anti-patrones frecuentes

Lo que no hay que hacer. Si te suena familiar, refactorizá.

⚠ God Composable

Una sola función @Composable de 400 líneas que mezcla estado, lógica, llamadas a red, validaciones y UI. Solución: descomponer en composables más chicos y mover la lógica al ViewModel.

⚠ ViewModel obeso

Un ViewModel que hace SQL, llamadas HTTP, valida reglas de negocio y formatea texto para UI. Solución: extraer a Repository, UseCase y mappers.

⚠ Repository anémico

Una clase Repository que solo reenvía llamadas a la API sin agregar nada. Solución: o agregás cache/sincronización/políticas, o evaluás si aporta valor mantenerla.

⚠ Estado duplicado

Mismo dato en remember, en Room, en una variable de la API y en el ViewModel. Inconsistencias garantizadas. Solución: Single Source of Truth.

⚠ Side effects sin control

viewModel.cargarDatos() escrito directamente en el body de un composable. Se ejecuta en cada recomposición → loops, requests duplicados, etc. Solución: LaunchedEffect o llamadas desde init { } del ViewModel.

⚠ NavController inyectado en todo

Pasar NavController a todos los composables hijos genera acoplamiento. Solución: pasar callbacks específicos como onBack, onLoginOk, onDetalleClick.

Persistencia de datos

Las opciones para guardar datos en Android, ordenadas de menos a más capaces.

Android ofrece muchas opciones de persistencia porque no todos los datos tienen las mismas necesidades. La pregunta correcta no es "¿cuál es mejor?" sino "¿mejor para qué?".

Las opciones, en orden creciente de complejidad

Memoria · remember
Estado temporal de pantalla. Se pierde al cerrar la app.
SharedPreferences
Clave-valor simple. Histórico, simple, pero síncrono y sin Flow.
DataStore
Reemplazo moderno de SharedPreferences: coroutines, Flow, transaccional.
Archivos internos
Privados de la app, eliminados al desinstalar.
Archivos externos
Compartibles vía MediaStore o Storage Access Framework.
SQLite directo
Base relacional, potente pero verbose. Riesgo de SQL injection.
Room
Capa moderna sobre SQLite. La opción local recomendada.
Backend
Para datos compartidos entre dispositivos, sincronización, lógica centralizada.

SharedPreferences vs DataStore

SharedPreferences es el mecanismo histórico para clave-valor. Funciona, pero es síncrono y no integra bien con coroutines/Flow.

DataStore es la evolución moderna. Funciona con coroutines y Flow, se integra naturalmente con Compose:

val Context.dataStore by preferencesDataStore(name = "settings")

class SettingsRepository(private val context: Context) {
    private val DARK_MODE = booleanPreferencesKey("dark_mode")

    val darkMode: Flow<Boolean> = context.dataStore.data
        .map { preferences -> preferences[DARK_MODE] ?: false }

    suspend fun setDarkMode(enabled: Boolean) {
        context.dataStore.edit { it[DARK_MODE] = enabled }
    }
}

// En Compose:
val darkMode by settingsRepository.darkMode.collectAsState(initial = false)
⚠ Error común

Guardar objetos completos serializados como JSON dentro de SharedPreferences. Técnicamente se puede pero no es recomendable para datos estructurados. Si aparecen listas, relaciones, búsquedas o filtros, la respuesta es Room.

SQL Injection · sí, también pasa en mobile

Si construís queries concatenando strings con entrada del usuario, sos vulnerable a SQL injection también en una app local. Reglas:

  • Nunca concatenar entrada del usuario en SQL.
  • Usar parámetros (? en SQLite, :param en Room).
  • Validar entradas, vengan de UI, intents, deep links, archivos importados, QR, NFC o backend.
  • Usar Room siempre que sea posible: las queries con @Query son parametrizadas por construcción.
// MAL: vulnerable
val cursor = db.rawQuery("SELECT * FROM usuarios WHERE email = '" + email + "'", null)

// BIEN: parametrizada
val cursor = db.rawQuery("SELECT * FROM usuarios WHERE email = ?", arrayOf(email))

// MEJOR: Room
@Query("SELECT * FROM usuarios WHERE email = :email")
suspend fun buscarPorEmail(email: String): UsuarioEntity?

Room · persistencia local moderna

Room es la opción recomendada de Jetpack para persistencia local estructurada. Se construye sobre SQLite y ofrece una capa más segura y cómoda. Trabaja con tres elementos:

Entity · representa una tabla

@Entity(tableName = "usuarios")
data class UsuarioEntity(
    @PrimaryKey(autoGenerate = true) val id: Int = 0,
    val nombre: String,
    val email: String

DAO · operaciones sobre la tabla

@Dao
interface UsuarioDao {
    @Query("SELECT * FROM usuarios")
    fun observarUsuarios(): Flow<List<UsuarioEntity>>

    @Insert
    suspend fun insertar(usuario: UsuarioEntity)

Database · contenedor

@Database(entities = [UsuarioEntity::class], version = 1)
abstract class AppDatabase : RoomDatabase() {
    abstract fun usuarioDao(): UsuarioDao

Repository híbrido (Room + Backend)

El patrón estrella para apps reales. La UI observa Room (siempre actualizado, funciona offline). El Repository sincroniza con la API en background.

class UsuarioOfflineFirstRepository(
    private val dao: UsuarioDao,
    private val api: UsuarioApi

    override fun observarUsuarios(): Flow<List<Usuario>> =
        dao.observarUsuarios().map { lista -> lista.map { it.toDomain() } }

        val creado = api.crearUsuario(usuario.toDto())

Ideas de los papers

Lo importante de cada paper de la cátedra, sin hundirse en los detalles.

Tres papers académicos fueron material del curso. No hace falta memorizar metodología ni estadísticas: alcanza con tener clara la idea central de cada uno y los conceptos transversales que aparecen en los tres.

Paper 1 · Aldayel & Alnafjan, 2017
Challenges and Best Practices for Mobile Application Development

Tesis central: el desarrollo mobile es distinto del desarrollo de software tradicional. Tiene restricciones específicas (energía, pantalla, ancho de banda variable, contexto cambiante, interrupciones constantes) y por eso necesita prácticas propias.

Lo que tenés que sacar:

  • Las tres categorías de apps: Native, Web, Hybrid, con sus pros/contras.
  • Las características que diferencian apps mobile del software tradicional (las que vimos en la sección 01).
  • Los desafíos: fragmentación de dispositivos, heterogeneidad de OS, multitasking limitado, form factors variados, time-to-market corto.
  • Best practices por fase del SDLC: planning con UX prototipado, requirements iterativos, testing en dispositivos reales (no solo emuladores), continuous testing, automation, testing "in the wild".
  • El enfoque Agile es el más apropiado para mobile por la naturaleza cambiante de los requisitos.
Paper 2 · Zein, Salleh & Grundy, 2016
A Systematic Mapping Study of Mobile Application Testing Techniques

Tesis central: testear apps móviles es distinto que testear apps de desktop o web. Los autores revisan 79 estudios y los clasifican en cinco subcategorías: usability testing, test automation, context-awareness, security, general.

Lo que tenés que sacar:

  • Las apps mobile requieren testear contexto además de inputs (sensores, GPS, red cambiante, ciclo de vida).
  • La mayoría de los frameworks de testing existentes (~64% de los estudios) apuntan a Android.
  • Falta investigación sobre life-cycle conformance (cómo verificar que la app respeta el ciclo de vida) y mobile services testing.
  • Herramientas como AndroidRipper automatizan exploración de la GUI; Robotium y MonkeyRunner son referentes en testing automatizado Android.
  • El testing manual sigue siendo lo más usado en la práctica; los frameworks unitarios típicos no soportan capacidades específicas de mobile (GPS, sensores).
Paper 3 · Holl & Elberzhager, 2016
Quality Assurance of Mobile Applications: A Systematic Mapping Study

Tesis central: es una mirada panorámica al quality assurance mobile (no solo testing). Analiza 230 papers según siete preguntas: tipos de QA, niveles de testing, fases del proceso, calidades atendidas, automatización, evaluación y desafíos.

Lo que tenés que sacar:

  • Niveles de testing: system (73%, mayoritario), unit (21%), integration (4%), independiente (3%).
  • Fases del proceso de testing (según ISTQB): planning & control, analysis & design, implementation & execution, evaluating exit criteria, test closure. La gran mayoría de los estudios atacan implementation & execution.
  • Calidades atendidas: ~59% se enfoca en funcionalidad; el resto en no-funcionales (security, usability, efficiency, compatibility, performance, reliability).
  • El topic de QA mobile creció fuerte desde 2003 y se acelera, especialmente en automation de GUI tests.
  • Conclusión: faltan estudios de evaluación rigurosos en industria, y faltan métricas estandarizadas.

Lo que comparten los tres papers

i Síntesis transversal
  • Mobile es diferente: requiere prácticas, herramientas y métricas propias.
  • Las restricciones físicas (batería, pantalla, red) son condicionantes de diseño y de testing.
  • El contexto (ubicación, sensores, red, energía) es entrada del programa, no solo escenario.
  • La fragmentación de dispositivos y OS es el mayor dolor de cabeza recurrente.
  • El testing en dispositivos reales es indispensable; los emuladores no alcanzan.
  • Agile y entregas frecuentes funcionan mejor que cascada para mobile.
  • Hay research gaps en testing de servicios móviles, life-cycle conformance, métricas estandarizadas y evaluación con industria.

Java & Kotlin · la relación

Por qué conviven, cómo interoperan y por qué Kotlin ganó.

Kotlin es un lenguaje JVM desarrollado por JetBrains. Google lo adoptó como lenguaje oficial de Android en 2017 y como preferido en 2019. La relación clave: Kotlin compila al mismo bytecode JVM que Java (.class), por eso ambos coexisten en un mismo proyecto Android.

Interoperabilidad bidireccional total

  • Una clase Kotlin puede heredar de una clase Java (y viceversa).
  • Un método Kotlin puede llamarse desde Java (y viceversa).
  • Las librerías Java del ecosistema Android histórico (la mayoría) se usan desde Kotlin sin wrappers.
.java (javac) ─┐ ├─→ .class (bytecode JVM) → D8 → .dex → ART .kt   (kotlinc) ─┘

Por qué Google empujó Kotlin

Más conciso
Menos boilerplate: data classes, properties, smart casts, expresiones en lugar de statements.
Null safety
El sistema de tipos distingue String? (nullable) de String (no-nullable). Atrapa NullPointerExceptions en compile-time.
Coroutines nativas
Base del código asíncrono moderno: suspend, Flow, StateFlow. En Java es mucho más verbose.
Funciones de extensión
Agregar métodos a clases existentes sin heredar.
Lambdas y HOF
Funciones de orden superior sin la verbosidad de Java.
100% interoperable
No hay que migrar todo: convivir es válido y común en proyectos legacy.

Ejemplo de interop

// Kotlin usando una clase Java estándar
val list: ArrayList<String> = ArrayList()
list.add("hola")

// Kotlin definiendo algo que Java puede usar
class Calculadora {
    fun sumar(a: Int, b: Int) = a + b
}
i Para el examen

Cuando preguntan "relación Java-Kotlin", la respuesta corta es: ambos compilan al mismo bytecode JVM, por eso interoperan al 100% en Android. Kotlin es la elección moderna por null safety, coroutines y concisión, pero el ecosistema Android histórico está en Java y Kotlin lo consume sin fricción.

MVVM y otras arquitecturas

El patrón arquitectónico de Compose, en detalle, y los alternativos.

MVVM = Model-View-ViewModel. Patrón arquitectónico que separa la UI de la lógica en tres capas:

Model
Datos y reglas de negocio: entidades, repositorios, fuentes de datos.
View
La UI (Composables). Solo describe qué mostrar y delega los eventos del usuario.
ViewModel
Intermediario. Mantiene el estado de UI, procesa eventos y coordina con el Model. Sobrevive recomposiciones y rotaciones.
View · Composable ↓ eventos     ↑ estado observable ViewModel · estado de UI + lógica de presentación ↓ Model · Repository / UseCase / DataSource

Por qué MVVM encaja perfecto con Compose

La View observa el estado del ViewModel; cuando el estado cambia, Compose recompone automáticamente. Es Unidirectional Data Flow puro. El ViewModel no conoce a la View — solo expone estado observable.

Ejemplo concreto · login

// MODEL
data class Usuario(val email: String, val nombre: String)

data class LoginUiState(
    val email: String = "",
    val password: String = "",
    val usuario: Usuario? = null,
    val error: String? = null
)

// VIEWMODEL
class LoginViewModel(private val repo: UsuarioRepository) : ViewModel() {
    private val _estado = MutableStateFlow(LoginUiState())
    val estado: StateFlow<LoginUiState> = _estado.asStateFlow()

    fun onEmailChange(email: String) {
        _estado.value = _estado.value.copy(email = email)
    }

    fun ingresar() {
        viewModelScope.launch {
            val usuario = repo.login(_estado.value.email, _estado.value.password)
            _estado.value = _estado.value.copy(usuario = usuario)
        }
    }
}

// VIEW
@Composable
fun LoginScreen(viewModel: LoginViewModel) {
    val estado by viewModel.estado.collectAsState()
    Column {
        TextField(value = estado.email, onValueChange = viewModel::onEmailChange)
        Button(onClick = viewModel::ingresar) { Text("Ingresar") }
    }
}

Otras arquitecturas comunes

MVC · Model-View-Controller

El clásico de los 70s. El Controller recibe el input del usuario, modifica el Model, y la View se actualiza. En Android puro pre-Compose, las Activities terminaban siendo Controllers que mezclaban todo (UI + lógica + acceso a datos). Difícil de mantener y testear.

MVP · Model-View-Presenter

La View es pasiva: el Presenter le dice explícitamente qué mostrar (vía interfaces). Era popular en Android pre-Compose. Diferencia clave con MVVM: el Presenter conoce la View y la actualiza directamente; el ViewModel no conoce la View y solo expone estado observable. MVVM es menos acoplado.

Clean Architecture

No es alternativa a MVVM, es complementaria. MVVM organiza la capa de presentación; Clean Architecture organiza toda la app en capas (UI → ViewModel → UseCase → Repository → DataSource) con la regla de dependencia: las capas internas no conocen las externas. Es lo que se usa en este curso.

MVI · Model-View-Intent (Redux-like)

Estado global inmutable, eventos como "intents" que producen un nuevo estado vía un reducer. Más estricto que MVVM, útil para apps con estado muy complejo. En Compose se ve cuando se usan sealed interface para representar todos los eventos posibles de una pantalla.

Resumen para el examen

MVC: Controller mezcla todo · acoplado. MVP: Presenter conoce la View · acoplado pero testeable. MVVM: ViewModel no conoce la View, expone estado observable · ideal para Compose y reactivo. Clean Architecture: organiza el resto de las capas, encima de cualquiera de los anteriores.

StateFlow

El "estado observable con valor actual" que conecta ViewModel y Compose.

StateFlow es un flujo (Flow) observable que siempre tiene un valor actual. Pertenece a las coroutines de Kotlin y es el mecanismo moderno para exponer estado desde un ViewModel.

Características clave

Hot
El flujo está activo aunque nadie esté observando. (Un Flow normal es "cold": se ejecuta cuando alguien lo collecta.)
Tiene valor inicial
Obligatorio. Siempre podés pedir .value y obtener algo.
Conflado
Solo emite cuando el valor realmente cambia. Setear el mismo valor dos veces no emite dos veces.
Thread-safe
Diseñado para lectura y escritura concurrente segura.

Patrón típico en ViewModel

Privado mutable, público read-only:

class UsuariosViewModel : ViewModel() {
    private val _estado = MutableStateFlow<UsuariosUiState>(UsuariosUiState.Loading)
    val estado: StateFlow<UsuariosUiState> = _estado.asStateFlow()  // read-only

    fun cargar() {
        viewModelScope.launch {
            _estado.value = UsuariosUiState.Loading
            val data = repo.obtener()
            _estado.value = UsuariosUiState.Success(data)  // emite el nuevo valor
        }
    }
}

@Composable
fun Pantalla(vm: UsuariosViewModel) {
    val estado by vm.estado.collectAsState()  // observa
    when (estado) {
        is UsuariosUiState.Loading -> CircularProgressIndicator()
        is UsuariosUiState.Success -> Lista(estado.usuarios)
        is UsuariosUiState.Error -> Text(estado.mensaje)
    }
}

StateFlowSí (obligatorio)HotEstado de UI desde ViewModel
SharedFlowNo (configurable)HotEventos one-shot (navegar, snackbar)

Si preguntan "qué es un StateFlow", lo importante: (1) es un Flow observable con valor actual obligatorio, (2) es hot y conflado, (3) es la herramienta estándar para exponer estado desde un ViewModel a Compose, observado con .collectAsState().

¿Qué es un Composable?

Definición compacta para preguntas de examen.

Un composable es una función Kotlin marcada con la anotación @Composable que describe una porción de UI en función del estado actual.

Reglas fundamentales

  1. Lleva la anotación @Composable.
  2. Recibe datos por parámetro y describe UI; no devuelve valor (su retorno es Unit).
  3. Solo puede llamarse desde otro composable o desde setContent { }.
  4. Su cuerpo debe ser idempotente: ejecutarlo dos veces con los mismos parámetros produce la misma UI.
  5. Puede ejecutarse muchas veces (recomposición), por eso no debe contener lógica pesada ni side effects libres.
@Composable
fun TarjetaUsuario(nombre: String, edad: Int) {
    Card {
        Column(Modifier.padding(16.dp)) {
            Text(text = nombre, style = MaterialTheme.typography.titleMedium)
            Text(text = "Edad: $edad")
        }
    }
}

// Uso:
TarjetaUsuario(nombre = "Ana", edad = 25)

Lo que lo hace especial

El compilador de Compose genera código adicional que registra qué estado lee cada composable. Cuando ese estado cambia, Compose solo re-ejecuta los composables afectados, no toda la pantalla. Esa es la recomposición selectiva, y es lo que hace al modelo declarativo eficiente.

Definición de una línea

Un composable es una función @Composable que describe UI a partir de estado, puede recomponerse, y es la unidad básica de construcción de interfaces en Jetpack Compose.

Observer pattern

El patrón GoF que está debajo de todo el modelo reactivo de Android.

Un Observer es un patrón de diseño (uno de los GoF clásicos) donde un objeto (el observable o subject) notifica automáticamente a múltiples objetos suscriptos (los observers) cuando su estado cambia.

Observable / Subject │ notifyAll() ├──→ Observer 1 ├──→ Observer 2 └──→ Observer 3

Dónde aparece en Android

Casi nunca lo implementás "a mano", pero está en todas partes:

LiveData
Histórico. Llamabas liveData.observe(this) { ... } y el observer se notificaba al cambiar el valor.
StateFlow / Flow
Versión moderna basada en coroutines. collectAsState() o collect { } es la suscripción.
Compose
La recomposición es Observer a gran escala. Compose observa qué estado lee cada composable y lo re-ejecuta cuando cambia.
RxJava
Librería entera basada en Observer (Observable.subscribe(observer)). Pre-coroutines, todavía vigente en proyectos grandes.

Por qué importa el concepto: todo el modelo reactivo de Android moderno (UI = f(estado), UDF, MVVM con StateFlow) es una aplicación a gran escala del patrón Observer. Sin él no existiría Compose como lo conocés.

Implementación conceptual mínima

class MiObservable {
    private val observers = mutableListOf<(String) -> Unit>()

    fun observar(callback: (String) -> Unit) { observers.add(callback) }

    fun emitir(valor: String) { observers.forEach { it(valor) } }
}

val obs = MiObservable()
obs.observar { dato -> println("Recibí: $dato") }
obs.emitir("hola")  // imprime: Recibí: hola

Builder · Proxy · Adapter

Tres patrones GoF que usás en Android todo el tiempo, aunque no lo notes.

Los tres son patrones clásicos del libro Gang of Four (1994). Son los que aparecen con más frecuencia en código Android.

Builder

Problema: construir un objeto con muchos parámetros opcionales sin caer en constructores con 15 argumentos.

Idea: un objeto auxiliar (builder) que va recibiendo configuración paso a paso, y al final construye el objeto real.

En Android:

  • AlertDialog.Builder (clásico).
  • NotificationCompat.Builder.
  • Retrofit.Builder (configurar el cliente HTTP).
  • Room.databaseBuilder.
val notif = NotificationCompat.Builder(context, "canal")
    .setContentTitle("Hola")
    .setContentText("Mensaje")
    .setSmallIcon(R.drawable.ic_notif)
    .setAutoCancel(true)
    .build()  // ← acá construye el objeto final

Proxy

Problema: controlar el acceso a un objeto (lazy initialization, control de permisos, caching, logging, llamadas remotas) sin que el cliente sepa que está hablando con un intermediario.

Idea: un objeto que tiene la misma interfaz que el real, pero intercepta las llamadas.

En Android:

  • Retrofit genera proxies dinámicos de tus interfaces de API. Cuando llamás api.getUsuarios(), el proxy traduce eso a una request HTTP. Vos pensás que llamás a una función Kotlin; en realidad llamás a un proxy que hace la red.
  • Binder es esencialmente un proxy: cuando una app llama a LocationManager, está llamando a un proxy local que hace IPC al servicio real del sistema.
  • Hilt y muchos frameworks de DI usan proxies para inyectar dependencias.
interface UsuariosApi {
    @GET("usuarios")
    suspend fun obtener(): List<UsuarioDto>
}

// Retrofit te da un proxy:
val api: UsuariosApi = retrofit.create(UsuariosApi::class.java)
val usuarios = api.obtener()  // parece llamada local, en realidad hace HTTP

Adapter

Problema: hacer que dos interfaces incompatibles trabajen juntas. Convierte la interfaz de una clase en otra que el cliente espera.

En Android (clásico):

  • RecyclerView.Adapter: el caso más conocido. RecyclerView espera una interfaz fija (getItemCount, onCreateViewHolder, onBindViewHolder). Vos extendés Adapter para conectar tu lista de datos con esa interfaz.
  • ArrayAdapter para ListView y Spinner.
  • En arquitectura: mappers entre DTO ↔ Entity ↔ Domain también son adapters conceptuales.
class UsuariosAdapter(
    private val usuarios: List<Usuario>
) : RecyclerView.Adapter<UsuariosAdapter.ViewHolder>() {

    override fun getItemCount() = usuarios.size

    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int): ViewHolder {
        val view = LayoutInflater.from(parent.context)
            .inflate(R.layout.item_usuario, parent, false)
        return ViewHolder(view)
    }

    override fun onBindViewHolder(holder: ViewHolder, position: Int) {
        holder.bind(usuarios[position])
    }
}

LinearLayout vs ConstraintLayout

Son dos layouts del sistema clásico de Views (XML). Conviene tener clara la diferencia conceptual aunque uses Compose.

LinearLayout
Apila las vistas en una sola dirección: vertical u horizontal (android:orientation). Simple, predecible. Para layouts complejos requiere anidar varios LinearLayouts → jerarquía profunda → mala performance.
ConstraintLayout
Cada vista define su posición mediante restricciones (constraints) hacia otras vistas o hacia el padre. Permite layouts complejos en una sola jerarquía plana. Mejor performance.

Cuándo usar cada uno:

  • LinearLayout: listas verticales simples, formularios cortos, barras de botones horizontales. Un caso simple donde la jerarquía no se complica.
  • ConstraintLayout: layouts complejos con elementos interdependientes (avatar a la izquierda, nombre arriba a la derecha, descripción debajo, botón abajo a la derecha). Recomendado por Google como default.

Ejemplo hipotético LinearLayout (tres campos apilados verticalmente en un formulario):

<LinearLayout android:orientation="vertical">
    <EditText android:hint="Nombre" />
    <EditText android:hint="Email" />
    <Button android:text="Enviar" />
</LinearLayout>

Ejemplo hipotético ConstraintLayout (perfil con avatar, nombre y bio):

<ConstraintLayout>
    <ImageView android:id="@+id/avatar"
        app:layout_constraintTop_toTopOf="parent"
        app:layout_constraintStart_toStartOf="parent" />
    <TextView android:id="@+id/nombre"
        app:layout_constraintStart_toEndOf="@id/avatar"
        app:layout_constraintTop_toTopOf="@id/avatar" />
</ConstraintLayout>
⚠ Equivalente en Compose

En Compose, Column / Row reemplazan a LinearLayout, y ConstraintLayout existe como composable (ConstraintLayout { }) pero rara vez se usa: la composición de Column/Row/Box cubre el 95% de los casos.

AndroidManifest.xml

El AndroidManifest.xml es el archivo de declaración de toda app Android. El sistema operativo lo lee antes de ejecutar la app para entender qué es y qué necesita. Es el "DNI" de la aplicación.

Para qué sirve

  • Declara el nombre del paquete (identificador único de la app).
  • Declara los componentes de la app: Activities, Services, BroadcastReceivers, ContentProviders.
  • Declara los permisos que la app necesita (cámara, ubicación, internet).
  • Declara las features de hardware requeridas (cámara, NFC, sensores).
  • Declara qué versión de Android soporta (minSDK, targetSDK).
  • Declara intent filters: cómo otras apps pueden invocarla.

Secciones principales

<manifest package="com.ejemplo.app">

    <!-- 1. Permisos -->
    <uses-permission android:name="android.permission.INTERNET" />
    <uses-permission android:name="android.permission.CAMERA" />

    <!-- 2. Features de hardware -->
    <uses-feature android:name="android.hardware.camera" />

    <!-- 3. Application: configuración global -->
    <application
        android:icon="@mipmap/ic_launcher"
        android:label="Mi App"
        android:theme="@style/AppTheme">

        <!-- 4. Componentes (activities, services, etc.) -->
        <activity android:name=".MainActivity">
            <intent-filter>
                <action android:name="android.intent.action.MAIN" />
                <category android:name="android.intent.category.LAUNCHER" />
            </intent-filter>
        </activity>

    </application>
</manifest>

minSDK y targetSDK

Estos dos valores se configuran en el build.gradle y se reflejan en el manifest (o vienen automáticamente). Definen el rango de versiones Android que tu app soporta.

minSdk
La versión más vieja de Android donde la app puede instalarse. Si lo bajás, llegás a más usuarios pero no podés usar APIs nuevas. Si lo subís, podés usar APIs modernas pero excluís dispositivos viejos.
targetSdk
La versión para la que la app está optimizada y testeada. Determina cómo se comporta el sistema con tu app. Google Play exige que sea relativamente reciente para nuevas publicaciones.

Implicancias de bajar el minSdk:

  • + Más alcance: más usuarios pueden instalar la app.
  • − Tenés que evitar APIs nuevas o usar compatibilidad (AndroidX).
  • − Más testing: hay que probar en versiones viejas.
  • − Bugs y comportamientos antiguos a tener en cuenta.

Implicancias de subir el minSdk:

  • + Acceso a APIs modernas sin checks de versión.
  • + Menos código de compatibilidad.
  • + Mejor seguridad y performance.
  • − Excluís dispositivos viejos: menos usuarios potenciales.

Heurísticas de Nielsen

Jakob Nielsen propuso en 1994 10 heurísticas de usabilidad: principios generales que toda interfaz debería cumplir. Aparecen en exámenes de UX. Las más útiles para mobile:

  1. Visibilidad del estado del sistema: el usuario siempre debe saber qué está pasando. Loaders al cargar, indicadores de progreso, confirmaciones al guardar.
  2. Concordancia con el mundo real: usar lenguaje del usuario, no jerga técnica. Iconos reconocibles. Una "papelera" para borrar, no "DEL ENTITY".
  3. Control y libertad del usuario: posibilidad de deshacer, cancelar, volver atrás. Salidas de emergencia claras.
  4. Consistencia y estándares: respetar las convenciones de la plataforma. En Android, seguir Material Design. Botones de "Aceptar/Cancelar" siempre en el mismo lugar.
  5. Prevención de errores: mejor evitar el error que mostrarlo. Validar inputs en tiempo real, deshabilitar botones cuando no aplican, pedir confirmación para acciones destructivas.
  6. Reconocimiento antes que recuerdo: mostrar opciones visibles en lugar de exigir que el usuario las recuerde. Menús, autocompletados, historiales.
  7. Flexibilidad y eficiencia: atajos para usuarios expertos, valores por defecto razonables para nuevos usuarios.
  8. Diseño estético y minimalista: cada elemento extra compite por la atención. Eliminar lo no esencial.
  9. Ayuda al usuario a reconocer y recuperarse de errores: mensajes de error en lenguaje claro, sugerir solución concreta. No solo "Error 404", sino "No se pudo conectar; revisá tu conexión".
  10. Ayuda y documentación: si hace falta, que sea fácil de buscar y orientada a tareas.
Para mobile, las 3-4 críticas

Si te piden "tres heurísticas clave para mobile", elegí visibilidad del estado del sistema (pantallas chicas, todo cambia rápido), prevención de errores (touch es impreciso, fácil tocar mal), consistencia (Material Design) y diseño minimalista (espacio limitado).

Wireframe

Un wireframe es un esquema visual de baja fidelidad de una pantalla. No incluye colores ni detalles estéticos: solo la estructura, la jerarquía de elementos y el flujo de información. Es el "plano arquitectónico" antes de pintar las paredes.

Para qué sirve:

  • Validar la estructura de información antes de invertir en diseño visual.
  • Comunicar ideas con cliente o equipo sin distraerse con la estética.

Diseño basado en contexto

El diseño basado en contexto (context-aware design) es la idea de que una app móvil debería adaptar su comportamiento a la situación real del usuario: ubicación, hora, conectividad, sensor data, batería, idioma, modo claro/oscuro.

El concepto es central a mobile porque, a diferencia del PC, el contexto cambia constantemente: el usuario puede estar en casa con Wi-Fi, en la calle con 3G, en el subte sin señal, todo en 10 minutos.

Ejemplos prácticos:

  • App de transporte que muestra la parada más cercana usando GPS.
  • App de fotos que sube automáticamente cuando hay Wi-Fi (no en datos).
  • App que reduce frecuencia de sincronización si la batería está baja.
  • App de noticias que cambia el idioma según la región del dispositivo.
  • App de delivery que muestra restaurantes abiertos a esa hora.

Caso hipotético detallado · app de running:

  • Contexto detectado: GPS para trazar ruta, acelerómetro para detectar pasos, hora del día, clima, batería del dispositivo.
  • Adaptación: si está oscureciendo, sugerir terminar la ruta; si la batería está baja, reducir actualización de GPS; si llueve, recomendar terminar; si el usuario corre habitualmente a la mañana, mostrar widget motivacional al despertar.
  • UX resultante: la app se siente "atenta" al usuario, no genérica. Reduce esfuerzo cognitivo: el usuario no tiene que configurar nada manualmente.

Arquitectura Java / JVM

"Arquitectura Java" en el contexto del examen se refiere a cómo se ejecuta el código Java, que es la base sobre la que se construye Android.

Código fuente .java ↓ javac (compilador) Bytecode .class (independiente de plataforma) ↓ se ejecuta sobre JVM · Java Virtual Machine ↓ traduce a código nativo CPU del dispositivo (cualquier OS / arquitectura)

Beneficios de la máquina virtual

  • "Write once, run anywhere": el mismo bytecode corre en cualquier sistema con una JVM compatible (Windows, Linux, macOS).
  • Aislamiento: el código corre en un entorno controlado. Más seguro que código nativo directo.
  • Gestión automática de memoria: garbage collector libera memoria sin intervención del programador.
  • Optimizaciones en runtime: la JVM puede aplicar JIT, profiling, optimizaciones adaptativas.
  • Portabilidad: la misma app no requiere recompilación para distintos OS.

JIT (Just-In-Time) en una oración

JIT compila el bytecode a código nativo durante la ejecución, justo en el momento en que se necesita. Identifica métodos "calientes" (muy usados) y los traduce a código máquina del procesador para que las próximas llamadas sean directas y rápidas. Combina las ventajas del bytecode portable con la velocidad del código nativo.

i Conexión con Android

Android no usa una JVM estándar. Usa ART, que entiende un bytecode propio (.dex, no .class). Pero el modelo conceptual es el mismo: bytecode intermedio + runtime que lo traduce a nativo. Por eso entender la arquitectura Java te da el marco para entender ART.

Asincronismo y APIs REST

Aparece como pregunta recurrente. Resumen rápido para responder:

Asincronismo es la capacidad de iniciar una operación que tarda (red, disco, cálculo) sin bloquear el hilo principal. La operación se ejecuta en otro hilo o de forma cooperativa, y cuando termina notifica el resultado.

Por qué se usa para consumir APIs: en Android, el hilo principal es el UI Thread. Una llamada HTTP puede tardar segundos. Si la hacés en el UI Thread, la app se congela y el sistema muestra un ANR (Application Not Responding). Toda llamada a una API debe ser asíncrona.

Librerías típicas para consumir API en Android:

Retrofit
Cliente HTTP de alto nivel. Convierte interfaces Kotlin en llamadas a la API.
OkHttp
Cliente HTTP de bajo nivel sobre el que corre Retrofit. Maneja conexiones, caché, interceptors.
Gson / Moshi / kotlinx.serialization
Parser JSON ↔ objetos Kotlin. Convierten la respuesta de la API en data classes.
Coroutines (kotlinx.coroutines)
Manejo de asincronismo en Kotlin. suspend, launch, viewModelScope.

Ejemplo · explicar un fragmento de data class consumido desde una API:

data class Correo(
    val id: Int,
    val Asunto: String,
    val Mensaje: String,
    val Adjunto: Boolean
)

Lo que ese fragmento dice:

  • data class: clase Kotlin especial pensada para contener datos. El compilador genera automáticamente equals(), hashCode(), toString() y copy().
  • Cuatro propiedades inmutables (val): id: Int, Asunto: String, Mensaje: String, Adjunto: Boolean.
  • Es candidata a ser un DTO (Data Transfer Object): el modelo que refleja el JSON devuelto por la API. Una librería como Moshi o Gson convierte automáticamente el JSON en una instancia de esta clase.

Ejemplo de uso completo con Retrofit:

interface CorreoApi {
    @GET("correos")
    suspend fun obtener(): List<Correo>
}

class CorreoViewModel(private val api: CorreoApi) : ViewModel() {
    private val _estado = MutableStateFlow<List<Correo>>(emptyList())
    val estado: StateFlow<List<Correo>> = _estado.asStateFlow()

    fun cargar() {
        viewModelScope.launch {
            _estado.value = api.obtener()
        }
    }
}

HCI · Interacción Humano-Computadora

La HCI (Human-Computer Interaction) es el campo que estudia cómo las personas interactúan con sistemas computacionales y cómo diseñarlos para que sean efectivos, eficientes y satisfactorios. Es multidisciplinario.

Psicología cognitiva

Ergonomía

Diferencia y relación entre UI / UX / CX:

  • UI es la capa visual: lo que ves y tocás. Color, tipografía, layout, iconos.
  • UX es la experiencia completa de uso: usabilidad, flujos, arquitectura de información, lógica de interacción. UX contiene a UI.
  • CX es la experiencia total con la marca: marketing, ventas, postventa, soporte, branding. CX contiene a UX.

Relación: UI ⊂ UX ⊂ CX. UI es una parte de UX, que es una parte de CX. La HCI investiga el cruce de todas estas áreas con el factor humano.

UX Research · desafíos

Vimos en la sección 08 qué es UX Research. Para examen, los desafíos típicos a los que se enfrenta el investigador:

  • Sesgos del usuario: la gente dice una cosa y hace otra. Hay que observar comportamiento real, no solo declaraciones.
  • Reclutamiento representativo: conseguir participantes que reflejen al usuario real, no a una conveniencia muestral.
  • Limitaciones de tiempo y presupuesto: la investigación profunda lleva semanas; el negocio quiere resultados ya.
  • Resistencia organizacional: convencer a stakeholders de que vale la pena invertir antes del desarrollo.
  • Validez ecológica: lo que pasa en un test de usabilidad en laboratorio no siempre refleja el uso real "en la calle".
  • Diversidad cultural: comportamientos cambian por país, edad, nivel socioeconómico. Difícil generalizar.
  • Síntesis de hallazgos: convertir 20 horas de entrevistas en insights accionables sin perder información clave.

Design Thinking · proceso de Stanford

El Stanford Design Thinking Process (SDTP) es la metodología más reconocida para innovación centrada en el ser humano. No es lineal: es un ciclo iterativo de 5 fases que se repite y ajusta durante todo el desarrollo. El objetivo es resolver problemas complejos construyendo productos que realmente satisfagan necesidades del usuario.

1. EMPATHIZE → 2. DEFINE → 3. IDEATE → 4. PROTOTYPE → 5. TEST ↑ │ └──────────────── (iteración) ─────────────────────────────┘
1 · Empathize
Ponerse en el lugar del usuario. Entender sus necesidades, deseos, miedos, motivaciones. Dejar de lado las suposiciones del equipo. Herramientas: entrevistas, encuestas, observación contextual (shadowing), análisis de datos. Postura: seek to understand, non-judgmental.
2 · Define
Destilar los hallazgos en un problema claro y accionable. En lugar de "la app es lenta", llegar a algo como "los usuarios se frustran al subir fotos porque la carga es lenta en redes móviles". Salidas típicas: personas, role objectives, decisions, challenges, pain points.
3 · Ideate
Brainstorming sin juzgar viabilidad. Generar la mayor cantidad de soluciones posibles, incluso las "locas". Compartir ideas, divergir/converger, pensar con "yes and...", priorizar. Aporte técnico: viabilidad de implementación.
4 · Prototype
Construir representaciones tangibles y de bajo costo de las ideas más prometedoras. Bocetos en papel, wireframes digitales, simulaciones interactivas. "Fail fast, iterate quickly", "keep it simple". El objetivo NO es la perfección, es la validación rápida.
5 · Test
Probar prototipos con usuarios reales. Identificar puntos de fricción. Los hallazgos pueden mandar al equipo de vuelta a fases anteriores (Empatizar/Definir). Decisiones basadas en datos, no en gusto personal.
⚡ Idea rectora

Design Thinking NO es una serie de pasos rígidos; es un ciclo. Si en el Test descubrís que el problema estaba mal definido, volvés a Define o Empathize. Iterar es el corazón del proceso. Por eso "fail fast, iterate quickly".

Ley de Fitts

Ley empírica de la HCI: el tiempo necesario para alcanzar un objetivo depende de la distancia hasta el objetivo y del tamaño del mismo.

T = a + b · log2(D/W + 1)

T = tiempo de movimiento · D = distancia al objetivo · W = ancho del objetivo
a, b = constantes empíricas que dependen del dispositivo (mouse, joystick, dedo)

El tiempo aumenta logarítmicamente con la distancia, pero disminuye también logarítmicamente a medida que el objetivo se hace más grande. Implicancias prácticas para diseño mobile:

Implicancias 1 · objetivos grandes son más rápidos

Botones, íconos y elementos importantes (Comprar, Enviar, CTA principales) deben ser grandes para reducir el tiempo de interacción. Los elementos pequeños frustran y aumentan errores.

Implicancias 2 · objetivos cercanos son más rápidos

La distancia es clave. Las acciones frecuentes y los elementos de menú deben colocarse cerca del punto de atención del usuario (área del pulgar en mobile). Minimizar el desplazamiento del cursor o del dedo.

Implicancias 3 · bordes y esquinas son "infinitos"

Un objetivo en el borde o esquina de la pantalla es el más rápido de alcanzar: el puntero/dedo no puede ir más allá. Por eso las esquinas son lugares privilegiados para colocar acciones primarias o navegación (ej: el botón de menú arriba a la izquierda, el FAB abajo a la derecha).

Implicancias 4 · menús radiales y pop-ups son eficientes

Una vez que se abre un menú, todos sus elementos están a la misma distancia corta del cursor → la selección es muy rápida. Por eso los menús contextuales que aparecen "donde el dedo ya está" son tan efectivos.

⚠ Aplicación a Android · tamaño mínimo de touch target

Material Design recomienda 48 dp × 48 dp (≈ 9 mm) para touch targets. iOS sugiere 44 × 44 pt. Estas medidas vienen directamente de la Ley de Fitts aplicada a la imprecisión del dedo. Botones más chicos = más errores = peor UX.

Diseño en función del contexto · Mobile UI

Vimos arriba el concepto general; en mobile UI hay aspectos específicos del contexto que el diseño debe contemplar:

  • Condiciones ambientales: iluminación (sol, oscuridad), ruido, distracciones. Afectan la legibilidad y la atención disponible.
  • Tareas y objetivos: por qué el usuario abrió la app en este momento. Información rápida, compra, comunicación, juego. Muy distinto el diseño según el caso.
  • Conexión de red: calidad y velocidad afectan tiempos de carga y responsividad. Diseñar para "lo peor".
  • Disponibilidad de datos: usuario con plan limitado vs Wi-Fi. Influye en si se cargan o no imágenes pesadas, video, etc.
  • Interacción física: gestos, voz, teclado, biometría. Mobile soporta varios; conviene aprovecharlos donde haga sentido.
  • Privacidad y seguridad: datos sensibles del usuario en un contexto donde "todos pueden ver la pantalla" (transporte público).

Contexto y entorno · usabilidad práctica

Los usuarios mobile están en movimiento, distraídos, usando una sola mano. La app debe ser usable:

  • en situaciones de uso rápido (esperando un colectivo, en una fila),
  • bajo diferentes condiciones de luz,
  • con una sola mano (típicamente el pulgar derecho).

Soluciones de diseño:

  • Optimizar la legibilidad (contraste alto, tipografía clara, tamaños suficientes).
  • Garantizar que los elementos interactivos sean accesibles con el pulgar (zona inferior de la pantalla).
  • Minimizar pasos para completar una tarea (reducir fricción).
  • Soportar modo oscuro y temas adaptativos.

Esto se conoce como diseño centrado en el usuario: parte de la comprensión profunda de las necesidades y expectativas del usuario en sus diferentes contextos.

Interacción y entrada · diseño táctil

La transición del mouse y teclado a la interacción táctil introduce nuevos retos:

Problema: los dedos son menos precisos que un cursor. Los elementos pequeños son difíciles de seleccionar, las interacciones complejas se vuelven engorrosas.

Solución:

  • Diseñar para el tamaño del dedo: área mínima de toque ~44×44 px (iOS) o 48×48 dp (Android).
  • Uso intuitivo de gestos: tocar, deslizar (swipe), pellizcar (pinch), arrastrar (drag).
  • Feedback visual claro para confirmar la acción del usuario (animación de tap, ripple en Material).

Accesibilidad · principios para mobile

Diseñar para todos los usuarios incluye discapacidades visuales, auditivas, motoras y cognitivas. Los principios fundamentales:

  • Contraste suficiente: relación de contraste mínima 4.5:1 para texto normal, 3:1 para texto grande (WCAG AA). Probar el diseño con simuladores de daltonismo.
  • Etiquetas adecuadas: cada elemento interactivo debe tener una descripción clara para lectores de pantalla (en Compose, contentDescription).
  • Soporte para lectores de pantalla (TalkBack en Android, VoiceOver en iOS). Estructura semántica clara.
  • Tamaño de texto escalable: respetar el tamaño que el usuario configura en el sistema. Usar sp (no dp) en texto.
  • Touch targets grandes: mínimo 48×48 dp incluso para elementos visualmente pequeños.
  • No depender solo del color: si un error se indica solo en rojo, un usuario daltónico no lo percibe. Combinar color con icono y/o texto.
  • Animaciones reducibles: respetar la preferencia del sistema "reducir movimiento" (algunos usuarios sufren mareos con animaciones).

C · Arquitectura de procesadores · RISC vs CISC, ARM, SoC

Entender la arquitectura para la cual desarrollás permite ganar performance y optimizar consumo. Es un diferencial concreto, no abstracto. Toda la industria mobile está construida sobre arquitectura ARM, y entender por qué te ayuda a tomar mejores decisiones técnicas.

Las dos filosofías: RISC y CISC

El mundo de los microprocesadores está dominado por dos filosofías:

CISC · Complex Instruction Set Computing
Pocas instrucciones por programa, pero cada instrucción es compleja y puede ocupar varios ciclos de reloj. Diseñado para reducir el código que escribe el programador. Ejemplos clásicos: IBM System/360, Intel 8086, Motorola 6800.
RISC · Reduced Instruction Set Computing
Muchas instrucciones simples y uniformes; cada una se ejecuta típicamente en un solo ciclo de reloj. Surge en los 80s al notar que los compiladores usaban un set acotado de instrucciones, y que con instrucciones simples se obtenía mejor rendimiento.

Mantra RISC clásico: "Don't add complexity unless it pays for itself" (no agregues complejidad a menos que se justifique).

ARM · el ganador en mobile

ARM (Advanced RISC Machines) es la arquitectura RISC que domina el mercado mobile al 99%. Características clave:

  • Es RISC puro: instrucciones simples, uniformes, optimizadas para pipeline.
  • ARM Ltd. no fabrica chips: licencia el diseño a fabricantes (Apple, Samsung, Qualcomm). Esto permitió que se convierta en estándar.
  • Cada fabricante diseña su propio SoC sobre el set ARM (Apple Silicon, Snapdragon, Exynos, Kirin).
  • Más de 250.000 millones de chips ARM fabricados desde su creación. Es la arquitectura "que no vemos pero nos acompaña día a día".

Por qué ARM ganó en mobile

Tres razones que se refuerzan mutuamente:

  1. Bajo consumo de batería. Por diseño, instrucciones simples consumen menos energía. Crítico cuando la batería es el recurso más escaso.
  2. Costo reducido de licencias. ARM licencia el set; los fabricantes diseñan sus chips. Más jugadores, más innovación, costos competitivos.
  3. Flexibilidad para integrarse en SoC. Combinable con módems, GPU, DSP, NPU en un único chip — exactamente lo que un smartphone necesita.

SoC · System on Chip

Un SoC es un circuito integrado que contiene los componentes de una computadora completa en un único sustrato de silicio. Es la pieza que hace posible la electrónica mobile moderna.

Un SoC típico de smartphone integra:

CPU (cores ARM) — lógica general GPU — gráficos, juegos, UI DSP — audio, cámaras, multimedia NPU / AI Engine — Machine Learning ISP — fotografía y video Modem 4G/5G — conectividad Sensores, controladores de energía Seguridad (TrustZone)

Ventajas de los SoC

  • Tamaño: del tamaño de una moneda. Cada vez más pequeños y más potentes.
  • Bajo consumo: optimizado para low-power → mayor duración de batería.
  • Flexibilidad: reprogramables y adaptables.
  • Confiabilidad: diseño relativamente sencillo, seguros.
  • Costo-eficiente: más económico que adquirir componentes por separado.

Desventajas / consideraciones

  • El proceso de diseño es complejo: 6 meses a 1 año, donde la verificación demanda el 70% del tiempo.

Es el mecanismo de IPC (Inter-Process Communication) que permite que procesos se comuniquen de forma segura en Android. Habilita llamadas desde el Application Framework hacia los servicios del sistema, y de ahí a HAL y al kernel.
4. Diferencia entre JIT y AOT en una oración cada uno.
JIT compila en runtime los métodos "calientes" (arranque rápido, adaptativo, más uso de RAM). AOT compila antes de ejecutar (arranque inmediato, mejor batería, más espacio en disco). ART moderno combina ambos: JIT para calentar y AOT en background guiado por perfiles.

Es el compilador que transforma bytecode JVM (.class) a bytecode Android (.dex). Hace falta porque Dalvik/ART no entienden .class, solo .dex. R8 es la versión de producción: D8 + shrinking + optimization + obfuscation.
6. ¿Qué significa "UI = f(estado)" en Compose?
La interfaz que ve el usuario es una función pura del estado actual de la app. Cuando el estado cambia, Compose vuelve a ejecutar las funciones composables afectadas (recomposición) para reflejar el nuevo estado. La UI no se modifica imperativamente: se describe declarativamente.
7. ¿Para qué sirven remember y mutableStateOf?
mutableStateOf(x) crea un estado observable por Compose: cuando lo modificás, Compose dispara la recomposición. remember conserva ese valor entre recomposiciones (sin él, se reiniciaría cada vez que se ejecuta el composable).
8. ¿Qué es state hoisting y qué problema resuelve?
Consiste en sacar el estado de un composable hijo y llevarlo a su llamador. El hijo recibe value: T y onValueChange: (T) -> Unit. Resuelve el acoplamiento del hijo a su estado interno: se vuelve stateless, reutilizable y testeable. El padre decide dónde vive el estado (memoria, ViewModel, etc.).
9. ¿Qué es Unidirectional Data Flow?
El estado baja hacia la UI; los eventos suben hacia quien modifica el estado. La UI no toca el estado de negocio: emite eventos que el state holder procesa y produce un nuevo estado. Esto reduce los caminos posibles de cambio de estado y mejora la predictibilidad.
10. ¿Qué responsabilidades tiene el ViewModel y cuáles NO?
SÍ: mantener estado de UI, procesar eventos, lanzar corrutinas en viewModelScope, invocar repositorios o use cases, exponer estado observable a Compose. NO: contener lógica de infraestructura (SQL crudo, HTTP directo, manipulación de archivos). Esos detalles pertenecen a la capa de datos.
11. ¿Para qué sirve LaunchedEffect?
Para ejecutar una corrutina asociada al ciclo de vida del composable. Si el composable sale, la corrutina se cancela. Si la key cambia, se cancela la anterior y se lanza una nueva. Es la forma correcta de iniciar trabajo asíncrono desde un composable (en vez de llamarlo directamente en el body, lo que se ejecutaría en cada recomposición).
12. ¿Qué hace Scaffold?
Proporciona la estructura visual estándar de una pantalla Material: barra superior, contenido principal, FAB, snackbar, barra inferior. Maneja correctamente los espacios ocupados por las barras vía paddingValues, que hay que aplicar al contenido.
13. Diferencia entre Repository y DataSource.
El DataSource sabe cómo hablar con una fuente concreta (Room, Retrofit, DataStore). El Repository decide cómo combinar fuentes, resolver conflictos, mapear datos y exponer una API coherente al resto de la app. Repository es la abstracción; DataSource es el implementador.
14. ¿Por qué tener DTO, Entity y Domain Model separados?
Para que la lógica de la app no quede acoplada a detalles externos. DTO refleja el contrato del backend, Entity refleja la tabla de Room, Domain es el modelo limpio que usa la lógica. Si el backend cambia un nombre de campo, solo cambiás el mapper, no toda la app.
15. ¿Qué es Single Source of Truth y por qué importa?
Una parte del sistema es la fuente autorizada de un dato. Si tenés copias múltiples e independientes (UI, ViewModel, Room, API), aparecen errores de sincronización imposibles de debuggear. Definir una fuente de verdad simplifica el razonamiento sobre la app.
16. ¿Qué es Offline First?
Una estrategia donde la app está diseñada para funcionar sin conexión. La UI lee siempre desde la base local (Room) y el Repository sincroniza con la API en background. La sincronización es responsabilidad explícita del Repository, no de la UI.
17. ¿Para qué sirve la inyección de dependencias?
Para entregar a una clase los objetos que necesita en lugar de que los cree. Reduce acoplamiento (la clase depende de interfaces, no de implementaciones concretas), facilita pruebas (podés inyectar fakes) y centraliza la construcción de objetos. En Android se usa Hilt.
18. Nombrá tres categorías de apps móviles y dales un trade-off.
Native (mejor performance y acceso al hardware, pero una codebase por plataforma). Web (multiplataforma con una sola codebase, pero sin acceso pleno al hardware ni offline). Hybrid (codebase única con acceso parcial al hardware, pero performance intermedia y dependés del framework).
19. Mencioná tres mecanismos de seguridad de Android.
(Cualquier tres de:) Sandboxing (cada app en proceso aislado con UID propio), Permission System (declarativos + runtime), SELinux (MAC a nivel kernel), Verified Boot, Google Play Protect, Cifrado de almacenamiento (FDE/FBE), App Signing, TLS/HTTPS por default.
20. ¿Cuándo usarías Room y cuándo SharedPreferences?
SharedPreferences (o mejor, DataStore) para preferencias simples clave-valor (ej: dark mode on/off, idioma). Room para datos estructurados con relaciones, búsquedas, filtros, listas. Si pensás guardar un JSON con una lista en SharedPreferences, eso es señal de que necesitás Room.
21. ¿Qué quiere decir que un composable sea "stateless"?
Que no guarda estado internamente: recibe los datos por parámetro y emite eventos hacia afuera. Es reutilizable, testeable y previewable. Patrón típico: (value: T, onValueChange: (T) -> Unit). Los composables más reutilizables deberían ser stateless.

Punto 1. Mencione y explique al menos 3 capas de la arquitectura de Android. Explique de qué manera un desarrollador puede interactuar con las diferentes capas.

Punto 2. Describa y realice un diagrama orientativo del proceso de compilación de una aplicación Android con cada una de sus fases.

El proceso de compilación de una app Android es una secuencia bien definida de etapas:

  1. Compilación a bytecode JVM: kotlinc y javac generan archivos .class (bytecode JVM, stack-based).
  2. Desugaring: el compilador reescribe sintaxis moderna (lambdas, default methods, APIs nuevas) a una forma compatible con APIs Android antiguos.
  3. D8 / R8: convierte los .class en .dex (register-based, eficiente para móviles). R8 además hace shrinking, obfuscation y minificación para releases.
  4. Empaquetado de recursos: drawables, layouts XML, strings, etc., son compilados y empaquetados.
  5. Firmado: el contenido se firma criptográficamente con la clave del desarrollador.
  6. Empaquetado final: se genera un .apk (instalable) o .aab (Android App Bundle, formato preferido por Google Play).

Diagrama:

┌─────────────────┐
│ .kt / .java     │  código fuente
└────────┬────────┘
         ↓ kotlinc / javac
┌─────────────────┐
│ .class          │  bytecode JVM (stack-based)
└────────┬────────┘
         ↓ desugaring
┌─────────────────┐
│ .class compat.  │
└────────┬────────┘
         ↓ D8 / R8
┌─────────────────┐
│ .dex            │  bytecode Android (register-based)
└────────┬────────┘
         ↓ + recursos + manifest
┌─────────────────┐
│ APK / AAB       │  firmado
└────────┬────────┘
         ↓ instalación
┌─────────────────┐
│ ART Runtime     │  ejecuta vía JIT + AOT
└─────────────────┘

Una vez instalada, ART verifica el .dex, lo interpreta con JIT y compila AOT en background los métodos hot, generando archivos .oat (nativo) y .vdex.

Punto 3. Compare los layouts LinearLayout y ConstraintLayout. ¿Qué diferencias conceptuales existen entre ambos y cuándo es recomendable utilizar cada uno? Justifique y proporcione un ejemplo de uso hipotético de cada uno.

Ambos son layouts del sistema clásico de Views (XML) en Android.

LinearLayout:

  • Apila las vistas en una sola dirección (vertical u horizontal) según android:orientation.
  • Simple y predecible.
  • Para layouts complejos requiere anidar varios LinearLayouts → jerarquía profunda → mala performance.

ConstraintLayout:

  • Cada vista define su posición mediante restricciones (constraints) hacia otras vistas o hacia el padre.
  • Permite layouts complejos en una sola jerarquía plana → mejor performance.
  • Recomendado por Google como default actual.

Cuándo usar cada uno:

  • LinearLayout: cuando el layout es realmente lineal y simple (formularios cortos, barras de botones, listas verticales sencillas). Si lo único que hago es apilar elementos uno tras otro, no necesito complejidad.
  • ConstraintLayout: layouts complejos con elementos interdependientes, donde un elemento depende de la posición de otro (avatar a la izquierda, nombre arriba a la derecha, descripción debajo, botón abajo a la derecha en una sola jerarquía).

    <EditText android:hint="Email" />
    <Button android:text="Enviar" />
</LinearLayout>

    <ImageView android:id="@+id/avatar"

        app:layout_constraintStart_toStartOf="parent" />

        app:layout_constraintStart_toEndOf="@id/avatar"

Punto 4. Explique las diferencias entre JIT y AOT. Mencione ventajas y desventajas de cada uno y cuál es el modelo de runtime que hoy persiste a partir de Android 7.0 (Lollipop). Tener en cuenta el consumo de CPU, memoria y experiencia al usuario al elaborar la respuesta.

Aclaración: Android 5.0 es Lollipop; Android 7.0 es Nougat. La pregunta probablemente refiere al modelo híbrido que se asentó en Android 7+.

JIT (Just-In-Time): compila el bytecode en runtime, solo los métodos que se están usando. El código se interpreta y los métodos "calientes" se compilan a nativo y se cachean.

  • + CPU/memoria: menor consumo de disco; arranque rápido (no compila todo antes).
  • + UX: instalación rápida; adaptativo al uso real.
  • − CPU: pausas durante la compilación en runtime.
  • − Memoria: mayor uso de RAM para la cache de código.
  • − UX: latencia inicial hasta que la app "entra en calor".

AOT (Ahead-Of-Time): compila antes de ejecutar, típicamente al instalar. El bytecode se traduce a binario nativo guardado en disco.

  • + CPU: menos uso de CPU en runtime.
  • + UX: arranque inmediato y estable; mejor batería.
  • − Memoria/disco: más espacio ocupado por binarios.
  • − UX: instalación lenta; menos adaptativo al uso real.

Modelo actual (Android 7+) · híbrido: ART combina JIT y AOT progresivamente:

  1. Instalación rápida sin compilación pesada (solo verificación + quickening).
  2. Primera ejecución con JIT: la app arranca rápido y el runtime registra qué métodos son hot en un perfil.
  3. Compilación AOT en background (dex2oat) cuando el dispositivo está cargando e idle, guiada por el perfil. Genera archivos .oat (nativo) y .vdex.
  4. Profile-on-Cloud: Google distribuye perfiles típicos junto con las apps; el AOT se aplica desde el momento cero.

Resultado en términos del usuario: instalaciones rápidas (JIT inicial), mejor performance progresiva (AOT background), menor consumo de batería que JIT puro, y menor uso de disco que AOT total.

Punto 1. Explique y proporcione ejemplos de la UX Research y a qué desafíos se enfrentan los investigadores. ¿Qué áreas intervienen en la investigación interacción humano-máquina?

UX Research es el proceso sistemático de estudiar usuarios para entender sus comportamientos, necesidades, motivaciones y contexto. Su objetivo es reducir suposiciones y basar las decisiones de diseño en datos y empatía, no en intuiciones del equipo.

Beneficios: reduce riesgo (valida la necesidad antes de invertir), ahorra dinero (corregir un problema en prototipo es muchísimo más barato que en producción), aumenta ROI (productos que resuelven problemas reales tienen mejor retención).

Tipos y ejemplos:

  • Cualitativa (responde al por qué): entrevistas en profundidad, observación contextual, pruebas de usabilidad. Ejemplo: entrevistar a 8 usuarios sobre cómo organizan su agenda para diseñar una app de calendario.
  • Cuantitativa (responde al qué y cuánto): encuestas, A/B testing, analítica de uso. Ejemplo: medir qué % de usuarios abandona el checkout en cada paso.
  • Generativa (al inicio del proyecto): descubrir necesidades, definir oportunidades.
  • Evaluativa (durante o después): validar prototipos, encontrar problemas en versiones existentes.

Desafíos a los que se enfrentan los investigadores:

  • Sesgos del usuario: la gente dice una cosa y hace otra. Hay que observar comportamiento real, no solo declaraciones.
  • Reclutamiento representativo: conseguir participantes que reflejen al usuario real, no solo a quienes están a mano.
  • Limitaciones de tiempo y presupuesto: el research profundo lleva semanas; el negocio quiere resultados ya.
  • Resistencia organizacional: convencer a stakeholders de invertir antes de codear.
  • Validez ecológica: lo que pasa en un test de laboratorio no siempre refleja el uso real "en la calle".
  • Diversidad cultural: los comportamientos cambian por país, edad, nivel socioeconómico.
  • Síntesis de hallazgos: convertir 20 horas de entrevistas en insights accionables.

Punto 2. Desarrolle el concepto de diseño móvil basado en contexto y su importancia en el desarrollo de aplicaciones móviles. Describa detalladamente un caso hipotético en el que se aplique este enfoque, destacando la UX y experiencia al usuario.

El diseño basado en contexto (context-aware design) es la idea de que una app móvil debería adaptar su comportamiento a la situación real del usuario: ubicación, hora, conectividad, sensor data, batería, idioma, modo claro/oscuro.

Importancia en mobile: a diferencia del PC, el contexto cambia constantemente. El usuario puede estar en casa con Wi-Fi, en la calle con 3G, en el subte sin señal, todo en 10 minutos. Una app que ignora el contexto da una experiencia genérica y frustrante; una que lo aprovecha se siente "atenta" al usuario.

Caso hipotético detallado · app de delivery de comida:

Contexto detectado:

  • Ubicación GPS: dónde está el usuario en este momento.
  • Hora del día: si es mediodía, mostrar opciones de almuerzo; si es noche, opciones de cena/snacks.
  • Conectividad: si la red es lenta, mostrar imágenes en baja resolución.
  • Batería: si está baja (< 15%), pausar actualización en tiempo real del repartidor.
  • Histórico de pedidos: si suele pedir pizza los viernes, mostrarla destacada esos días.
  • Modo del sistema: tema oscuro automático según el sistema.

Adaptaciones del diseño:

  • Pantalla principal muestra solo restaurantes abiertos a esa hora y cerca de la ubicación actual.
  • El campo de dirección viene pre-llenado con la dirección detectada por GPS, con opción de cambiar.
  • Si la red es lenta, se simplifica la UI y se reducen imágenes.
  • Si es de noche y el celular está en modo oscuro, la app respeta ese modo automáticamente.
  • Si llueve (vía API de clima), se sugieren entregas a domicilio en vez de retiro en local.

UX resultante:

  • Reducción de esfuerzo cognitivo: el usuario no configura nada manualmente; la app le presenta lo relevante.
  • Personalización percibida: "esta app me entiende".
  • Eficiencia: tiempo de pedido se reduce de minutos a segundos.
  • Confianza: sentirse acompañado, no obligado a configurar.

Trade-offs a considerar: hay que pedir permisos correctamente (GPS, notificaciones), respetar la privacidad (no usar más datos de los necesarios), y dejar al usuario controlar las adaptaciones (que pueda fijar dirección manualmente, por ejemplo).

Punto 3. Explique la arquitectura Java y realice un diagrama de alto nivel.
(a) ¿Qué beneficios presenta el uso de la máquina virtual?
(b) ¿Qué implica la compilación JIT (Just in time)?

La arquitectura Java se basa en un modelo de dos pasos: el código fuente se compila a un formato intermedio independiente de plataforma llamado bytecode, que luego se ejecuta sobre la JVM (Java Virtual Machine), la cual lo traduce al código nativo del procesador.

Diagrama de alto nivel:

┌─────────────────┐
│  .java          │  código fuente
└────────┬────────┘
         ↓ javac (compilador)
┌─────────────────┐
│  .class         │  bytecode (independiente de plataforma)
└────────┬────────┘
         ↓ se ejecuta sobre
┌─────────────────┐
│  JVM            │  Java Virtual Machine
│  ┌───────────┐  │
│  │ Intérprete│  │  ← lee bytecode
│  │ + JIT     │  │  ← compila métodos hot a nativo
│  │ + GC      │  │  ← gestiona memoria
│  └───────────┘  │
└────────┬────────┘
         ↓ código nativo
┌─────────────────┐
│  CPU del SO     │  Windows / Linux / Mac / Android (vía ART)
└─────────────────┘

(a) Beneficios de la máquina virtual:

  • "Write once, run anywhere": el mismo bytecode corre en cualquier sistema con una JVM compatible. No requiere recompilación para distintos OS.
  • Aislamiento y seguridad: el código corre en un entorno controlado. Más seguro que código nativo directo.
  • Gestión automática de memoria: el garbage collector libera memoria sin intervención del programador. Reduce errores comunes (memory leaks, dangling pointers).
  • Optimizaciones en runtime: la JVM puede aplicar JIT, profiling adaptativo, optimizaciones que un compilador AOT estático no puede hacer.
  • Robustez: errores en una clase no derrumban la VM; manejo estructurado de excepciones.

(b) Qué implica la compilación JIT:

JIT (Just-In-Time) implica compilar el bytecode a código nativo durante la ejecución, justo en el momento en que se necesita.

  • El bytecode se interpreta inicialmente.
  • La JVM hace profiling: registra qué métodos se ejecutan más frecuentemente (los "hot methods").
  • Esos métodos calientes se compilan a código máquina del procesador y se cachean.
  • Las próximas llamadas a esos métodos van directamente al nativo, sin interpretación.
  • La JVM puede incluso re-optimizar en función de cómo se usa el programa.

Ventaja clave: combina la portabilidad del bytecode con la velocidad del código nativo. Costo: uso de RAM para la cache y pausas momentáneas durante la compilación.

Conexión con Android: Android no usa una JVM estándar. Usa ART, que entiende un bytecode propio (.dex). Pero el modelo conceptual es el mismo: bytecode intermedio + runtime que lo traduce a nativo combinando JIT y AOT.

Punto 4. Explique el propósito del archivo AndroidManifest.xml en una aplicación Android. Describa sus secciones principales y la importancia de cada una. ¿Qué implica bajar o subir el minSDK?

El AndroidManifest.xml es el archivo de declaración de toda app Android. El sistema operativo lo lee antes de ejecutar la app para entender qué es y qué necesita. Es el "DNI" o documento de identidad de la aplicación.

Propósito principal:

  • Identificar la app (nombre del paquete, único en el ecosistema).
  • Declarar los componentes (Activities, Services, BroadcastReceivers, ContentProviders).
  • Declarar permisos requeridos (cámara, ubicación, internet, contactos).
  • Declarar features de hardware requeridas (cámara, NFC, sensores).
  • Declarar versiones de Android soportadas (minSdk, targetSdk).
  • Declarar intent filters: cómo otras apps o el sistema pueden invocar a la app.

Secciones principales:

<manifest package="com.ejemplo.app">

    <!-- 1. Permisos: lo que la app pide al usuario -->
    <uses-permission android:name="android.permission.INTERNET" />
    <uses-permission android:name="android.permission.CAMERA" />

    <!-- 2. Features: hardware necesario -->
    <uses-feature android:name="android.hardware.camera" />

    <!-- 3. Application: configuración global -->
    <application
        android:icon="@mipmap/ic_launcher"
        android:label="Mi App"
        android:theme="@style/AppTheme">

        <!-- 4. Componentes -->
        <activity android:name=".MainActivity">
            <intent-filter>
                <action android:name="android.intent.action.MAIN" />
                <category android:name="android.intent.category.LAUNCHER" />
            </intent-filter>
        </activity>

    </application>
</manifest>

Importancia de cada sección:

  • Permisos: el sistema (y el usuario) deciden si conceden lo que la app pide. Sin declararlos, la app crashea al intentar usar esas capacidades.
  • Features: indica al Play Store si la app es compatible con un dispositivo. Un celular sin cámara no debería ver una app que la requiere.
  • Application: definición global (icono, nombre visible, tema) que aplica a todos los componentes salvo override explícito.
  • Componentes y intent filters: el intent filter MAIN/LAUNCHER identifica qué Activity es el punto de entrada cuando se abre la app desde el launcher.

minSDK y targetSDK:

  • minSdk: la versión más vieja de Android donde la app puede instalarse.
  • targetSdk: la versión para la que está optimizada y testeada.

Implicancias de bajar el minSDK:

  • + Más alcance: más usuarios pueden instalar la app.
  • − No se pueden usar APIs nuevas (o requieren checks de compatibilidad).
  • − Más esfuerzo de testing en versiones viejas.
  • − Bugs y comportamientos antiguos a considerar.

Implicancias de subir el minSDK:

  • + Acceso a APIs modernas sin compatibilidad.
  • + Menos código de compatibilidad.
  • + Mejor seguridad (los Android viejos tienen vulnerabilidades sin parchar).
  • + Mejor performance.
  • − Excluís dispositivos viejos: menos usuarios potenciales.

Conclusión: bajar minSdk = más reach, menos features modernas. Subirlo = menos reach, más capacidades modernas. La elección depende del público objetivo y del nivel de fragmentación que se quiera tolerar.

Punto 5. Explique las diferencias clave entre el desarrollo de aplicaciones móviles nativas, híbridas y multiplataforma. Analice las ventajas y desventajas de cada enfoque en términos de rendimiento, costos de desarrollo, experiencia del usuario (UX) y mantenimiento. Proporcione ejemplos de tecnologías o frameworks utilizados en cada caso.

Toda app móvil se construye con uno de estos tres enfoques. La elección define capacidades, costo, performance y UX.

1. Nativas

Desarrolladas con el SDK oficial de cada plataforma. Una codebase por sistema operativo.

  • Tecnologías: Android → Kotlin/Java + Android Studio + Jetpack Compose / XML. iOS → Swift/Objective-C + Xcode + SwiftUI / UIKit.
  • Rendimiento: el mejor posible. Acceso directo a APIs y hardware.
  • UX: óptima. Cumple guidelines de cada plataforma (Material en Android, HIG en iOS). Animaciones fluidas.
  • Costo de desarrollo: alto. Dos equipos / dos codebases / dos lenguajes.
  • Mantenimiento: caro. Cada feature se implementa dos veces.
  • Cuándo conviene: apps que dependen del hardware (juegos, AR, cámara avanzada), apps premium donde la UX es diferenciador, apps con uso intensivo de sensores.

2. Híbridas

Combinan tecnologías web dentro de un wrapper nativo. La UI se renderiza en una WebView.

  • Tecnologías: Apache Cordova / PhoneGap, Ionic, Capacitor.
  • Rendimiento: bajo. La WebView agrega overhead, animaciones y scrolls se sienten "no nativos".
  • UX: aceptable para apps simples; deficiente para experiencias complejas. Dificultad para sentirse "Android" o "iOS" auténticos.
  • Costo de desarrollo: bajo. Una sola codebase web.
  • Mantenimiento: económico, una sola base de código.
  • Cuándo conviene: apps internas de empresa, MVPs, apps simples que ya existen como web y se quiere "envolver".

3. Multiplataforma (cross-platform)

Frameworks que generan UI nativa (no WebView) desde una codebase común.

  • Tecnologías: React Native (Meta, JavaScript/TypeScript), Flutter (Google, Dart), Kotlin Multiplatform Mobile (KMM, Kotlin), .NET MAUI / Xamarin (Microsoft, C#).
  • Rendimiento: bueno, cercano a nativo. Flutter renderiza directo al canvas; React Native usa puente JS-nativo.
  • UX: buena. Componentes nativos (RN) o renderizados consistentes (Flutter).
  • Costo de desarrollo: medio. Una codebase + ajustes específicos por plataforma.
  • Mantenimiento: medio. Dependencia del framework: si el framework no expone una API nueva, hay que escribir código nativo igual.
  • Cuándo conviene: startups que necesitan iOS y Android sin recursos para dos equipos, apps con UI relativamente estándar, productos donde el time-to-market es crítico.

Tabla comparativa:

                    Nativas        Híbridas        Multiplataforma
Rendimiento         Alto           Bajo            Medio-alto
UX                  Excelente      Aceptable       Buena
Costo desarrollo    Alto           Bajo            Medio
Mantenimiento       Caro           Económico       Medio
Acceso al HW        Total          Limitado        Bueno
Tecnologías         Kotlin/Swift   Cordova/Ionic   Flutter/RN/KMM

Cómo decidir:

  1. ¿Necesito hardware específico o máxima performance? → Nativa.
  2. ¿Tengo bajo presupuesto y la app es simple/interna? → Híbrida.
  3. ¿Quiero llegar a iOS y Android con un equipo chico, manteniendo UX buena? → Multiplataforma.

Punto 1. Explique el concepto de Jetpack Compose y su modelo declarativo. ¿Cuál es la idea central que lo gobierna? Mencione los componentes principales (composable, recomposición, estado, modifier) y proporcione un ejemplo de un composable simple que use estado.

Jetpack Compose es el toolkit moderno de Android para construir interfaces nativas en Kotlin. Reemplaza el modelo imperativo (XML + Views, donde se buscaban elementos y se modificaban paso a paso) por un modelo declarativo.

Idea central: UI = f(estado). La interfaz que ve el usuario es una función pura del estado actual de la app. Cuando el estado cambia, Compose vuelve a ejecutar las funciones afectadas (recomposición) para reflejar el nuevo estado.

Componentes principales:

  • Composable: función Kotlin con @Composable que describe UI. No devuelve valor; recibe datos por parámetro.
  • Recomposición: re-ejecución selectiva cuando cambia el estado observado. Solo se redibuja lo necesario.
  • Estado: remember { mutableStateOf(...) } para estado local; StateFlow desde un ViewModel para estado de pantalla.
  • Modifier: configura tamaño, padding, posición, interacción. Es la "navaja suiza" para customizar cualquier composable.

Ejemplo · contador con estado:

@Composable
fun Contador() {
    var n by remember { mutableStateOf(0) }
    Column(modifier = Modifier.padding(16.dp)) {
        Text("Valor: $n")
        Button(onClick = { n++ }) { Text("Incrementar") }
    }
}

Cada vez que se hace click, n++ modifica el estado. Compose detecta el cambio y recompone el Text que lee n. El Button no se recompone porque no depende de n.

Punto 2. Explique qué es la Inyección de Dependencias y por qué se usa en arquitectura mobile. Mencione un framework usado en Android. Proporcione un ejemplo donde se vea un ViewModel que recibe un Repository por inyección. Compare contra la alternativa "sin inyección" en términos de testabilidad y acoplamiento.

La Inyección de Dependencias (DI) es un patrón donde los objetos no crean internamente lo que necesitan, sino que les es entregado desde afuera. La clase declara qué necesita; otra capa decide cómo proveerlo.

Por qué se usa en mobile:

  • Reduce acoplamiento: la clase depende de interfaces, no de implementaciones concretas.
  • Facilita testing: se pueden inyectar implementaciones falsas (fakes) en tests unitarios.
  • Centraliza la construcción: la lógica de "cómo crear este objeto" vive en un solo lugar.
  • Permite cambiar implementaciones sin tocar el código que las usa (cambiar API real por una mock para QA, por ejemplo).

Framework usado en Android: Hilt (recomendado por Google, basado en Dagger). Otros: Koin (más liviano), Dagger directo.

Ejemplo SIN inyección (mal):

class LoginViewModel : ViewModel() {
    private val repository = UsuarioRepositoryImpl()  // creado a mano
}

Problemas: el ViewModel queda acoplado a una implementación concreta. No se puede testear con un repository falso. Si cambia el constructor del repo, hay que tocar el ViewModel.

Ejemplo CON inyección (bien, usando Hilt):

@HiltViewModel
class LoginViewModel @Inject constructor(
    private val repository: UsuarioRepository  // ← interfaz, no impl
) : ViewModel()

@Module
@InstallIn(SingletonComponent::class)
object AppModule {
    @Provides
    fun provideRepository(...): UsuarioRepository = UsuarioRepositoryImpl(...)
}

Comparación:

  • Acoplamiento: sin DI, alto (concreto); con DI, bajo (interfaz).
  • Testabilidad: sin DI, imposible inyectar fake; con DI, trivial.
  • Mantenibilidad: sin DI, cambios se propagan; con DI, cambios localizados al módulo de Hilt.

(c) ¿Qué ventajas ofrece este enfoque frente al uso directo de SQLite?
(d) ¿Cómo se usaría desde un ViewModel para mostrar la lista en una pantalla Compose?

(b) Anotaciones explicadas:

  • @Entity(tableName = "tareas"): marca la clase como una tabla de la base de datos. tableName define el nombre de la tabla en SQLite.
  • @PrimaryKey(autoGenerate = true): indica que id es la clave primaria y que SQLite generará automáticamente el siguiente número al insertar.

  • @Insert: genera automáticamente la operación de insert para esa entity, sin que tengas que escribir el SQL.
  • Flow<List<TareaEntity>>: el resultado se devuelve como un Flow observable: cuando algo cambia en la tabla, el Flow emite la lista actualizada automáticamente.
  • suspend: la función insertar es asíncrona; debe llamarse desde una corrutina para no bloquear el hilo principal.

(c) Ventajas frente a SQLite directo:

  • Verificación en compile-time: Room valida las queries cuando compilás. Si hay un error de sintaxis SQL o un campo mal escrito, no compila. Con SQLite directo, el error aparece en runtime.
  • Type safety: los resultados se mapean automáticamente a entities. No hay que manejar cursors a mano.
  • Queries parametrizadas por construcción: previene SQL injection.
  • Boilerplate generado: no hay que escribir el código de open/close de la base, ni el mapeo cursor-objeto.
  • Integración con coroutines y Flow: reactividad automática.
  • Migrations explícitas y manejables.
  • Testeable con in-memory database.

(d) Uso desde ViewModel + Compose:

class TareasViewModel(private val dao: TareaDao) : ViewModel() {

    val tareas: StateFlow<List<TareaEntity>> =
        dao.observarTodas()
           .stateIn(viewModelScope, SharingStarted.Eagerly, emptyList())

    fun agregar(titulo: String) {
        viewModelScope.launch {
            dao.insertar(TareaEntity(titulo = titulo, completada = false))
        }
    }
}

@Composable
fun ListaTareasScreen(vm: TareasViewModel) {
    val tareas by vm.tareas.collectAsState()
    LazyColumn {
        items(tareas) { tarea ->
            Text(text = tarea.titulo)
        }
    }
}

Cuando se inserta una tarea nueva, Room emite la lista actualizada en el Flow → el StateFlow se actualiza → la UI recompone automáticamente. Single Source of Truth + reactividad: la UI siempre refleja el estado real de la base.

Punto 4. Explique qué es una arquitectura limpia (Clean Architecture) aplicada a Android. Describa las capas que la componen, la regla de dependencia y proporcione un ejemplo del flujo completo desde un click del usuario en la UI hasta la obtención de un dato del backend.

Clean Architecture aplicada a Android organiza la app en capas concéntricas, cada una con una responsabilidad clara. Es complementaria a MVVM: MVVM organiza la capa de presentación; Clean Architecture organiza el resto.

Capas (de afuera hacia adentro):

  1. UI (presentación): composables. Renderizan estado y emiten eventos.
  2. ViewModel: state holder. Mantiene el estado de UI y procesa eventos. Coordina llamadas a use cases.
  3. Domain: UseCases y modelos puros. Reglas de negocio independientes de Android.
  4. Data: Repositories y DataSources (locales y remotos). Saben hablar con Room, Retrofit, DataStore.

Regla de dependencia: las capas internas no conocen a las externas. El dominio no debería importar Compose, Room ni Retrofit. Si tu UseCase importa androidx.compose.*, está mal acoplado.

Diagrama:

┌──────────────────────┐
│   UI (Composable)    │
└──────────┬───────────┘
           ↓ eventos / ↑ estado
┌──────────────────────┐
│   ViewModel          │
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│   UseCase (Domain)   │
└──────────┬───────────┘
           ↓
┌──────────────────────┐
│   Repository (Data)  │
└──────┬───────┬───────┘
       ↓       ↓
   Local     Remote
  (Room)    (Retrofit)

Flujo completo de un click · ejemplo "ingresar a la cuenta":

  1. El usuario tipea email/password y toca "Ingresar" en el composable LoginScreen.
  2. El composable llama viewModel.ingresar().
  3. El ViewModel ejecuta dentro de viewModelScope el LoginUseCase.
  4. El UseCase aplica reglas de negocio (validación de formato, longitud mínima de password) y llama al UsuarioRepository.
  5. El Repository decide la fuente: en este caso pide al RemoteDataSource.
  6. El RemoteDataSource usa Retrofit para hacer un POST /login; recibe el token.
  7. El Repository guarda el token en LocalDataSource (DataStore para preferencias) y devuelve un Usuario de dominio.
  8. El UseCase devuelve un Result<Usuario> al ViewModel.
  9. El ViewModel actualiza el StateFlow con el nuevo estado (Success o Error).
  10. El composable, que observa el StateFlow con collectAsState(), se recompone y muestra el resultado al usuario.

Ventajas:

  • Cada capa es testeable independientemente.
  • El dominio y los UseCases no dependen de Android: tests sin emulador.
  • Cambiar Retrofit por GraphQL solo afecta al RemoteDataSource.
  • La estructura es predecible: cualquiera del equipo sabe dónde buscar qué.
Punto 5. Una empresa quiere desarrollar una app móvil que muestre un catálogo de productos. La app debe funcionar tanto online como offline, sincronizar con un servidor cuando hay red, y mostrar inmediatamente los datos guardados localmente cuando no hay conexión. Diseñe la arquitectura proponiendo las capas, patrones y librerías que usaría. Justifique cada decisión.

El requerimiento describe un caso clásico de Offline First con Single Source of Truth. Diseño propuesto:

Capas y patrones:

  • UI (Compose): pantallas reactivas que observan estado del ViewModel. Patrón Screen-Content para separar lógica de presentación de renderizado.
  • ViewModel + StateFlow: expone UiState con loading/success/error como sealed interface.
  • UseCases: ObtenerProductosUseCase, SincronizarProductosUseCase, BuscarProductoUseCase.
  • Repository con estrategia Offline First.
  • LocalDataSource: Room (fuente de verdad).
  • RemoteDataSource: Retrofit + Moshi (sync con backend).

Librerías:

  • Jetpack Compose: UI declarativa.
  • Hilt: inyección de dependencias.
  • Room: persistencia local. Verificación en compile-time, Flow observable, queries parametrizadas (anti SQL injection).
  • Retrofit + OkHttp + Moshi: cliente HTTP moderno + parser JSON eficiente.
  • kotlinx.coroutines + StateFlow: asincronismo sin bloquear UI.
  • WorkManager: para tareas de sincronización periódica en background, resistentes a cierre de app y reinicios.

Flujo Offline First:

1. UI observa Repository.observarProductos() → Flow desde Room
   → muestra siempre lo que hay localmente

2. Al iniciar app o pull-to-refresh:
   Repository.sincronizar() →
     - Llama a Retrofit (RemoteDataSource)
     - Si OK: actualiza Room (LocalDataSource)
     - Room emite cambios → Flow → UI recompone

3. WorkManager dispara sincronización periódica
   (cada X horas, solo con red disponible y cargando)
   → llama a Repository.sincronizar() en background

Estructura del Repository:

class ProductoRepositoryImpl(
    private val local: ProductoLocalDataSource,
    private val remote: ProductoRemoteDataSource
) : ProductoRepository {

    override fun observarProductos(): Flow<List<Producto>> =
        local.observar().map { it.map { e -> e.toDomain() } }

    override suspend fun sincronizar() {
        try {
            val remotos = remote.obtener()
            local.reemplazarTodos(remotos.map { it.toEntity() })
        } catch (e: IOException) {
            // sin red: la UI sigue mostrando datos locales
        }
    }
}

Justificación de cada decisión:

  • Single Source of Truth (Room): la UI siempre lee de un solo lugar. Sin esto, habría inconsistencias entre lo que muestra la pantalla y lo que está realmente guardado.
  • Offline First: la app funciona aunque no haya red. La sincronización es responsabilidad del Repository, no de la UI.
  • Flow + StateFlow: la UI se actualiza automáticamente cuando cambian los datos. No hay que llamar manualmente a "refresh".
  • WorkManager: garantiza ejecución incluso si la app se cierra o el dispositivo se reinicia. Se ejecuta solo cuando hay red y batería suficiente, lo que cuida UX y consumo.
  • Hilt: facilita testear cada capa con fakes inyectados.
  • Compose: menos código y mejor performance que XML+Views; integración natural con StateFlow.
  • Mappers entre DTO/Entity/Domain: si cambia el formato del backend, solo se ajusta el mapper.

Resultado UX: el usuario ve los productos al instante (cache local), recibe actualizaciones cuando hay red, y la app no falla en el subte ni en zonas sin señal. Se siente fluida y confiable.

Punto 2. Mencione y explique al menos 3 capas de la arquitectura de Android. Explique de qué manera un desarrollador puede interactuar con las diferentes capas.

Android está organizado en capas. De arriba hacia abajo:

  1. Aplicaciones: las apps del usuario y del sistema. Se desarrollan con el SDK.
  2. Application Framework: clases de alto nivel que abstraen el hardware (ActivityManager, WindowManager, ViewSystem, TelephonyManager, LocationManager, ContentProviders). Es donde trabaja el desarrollador Android el 95% del tiempo.
  3. Android Runtime (ART) + librerías nativas C/C++: ART ejecuta el bytecode .dex (combina JIT y AOT). Las librerías incluyen OpenGL, WebKit, libc, accesibles vía NDK.
  4. HAL (Hardware Abstraction Layer): módulos .so en C/C++ que abstraen cada componente físico (cámara, GPS, BT). El framework los llama vía Binder.
  5. Linux Kernel: gestión de memoria, procesos, energía, drivers. Modificado y optimizado para móviles, mandatorio ARM/ARM64 desde Android 5.

Cómo interactúa el desarrollador con cada capa:

  • Capa de aplicaciones y framework: es la zona normal de trabajo, vía SDK + Kotlin/Java.
  • Capa de librerías C/C++: accesible mediante NDK + JNI, para apps que necesitan máxima performance (juegos, video, AR).
  • Capas de HAL y Kernel: el dev de apps no las toca. Solo intervienen fabricantes de hardware o desarrolladores de custom ROMs / firmware.

El dev escribe contra el framework; ese framework usa Binder IPC para hablar con servicios del sistema, que llaman al HAL, y el HAL al kernel.

Punto 3. ¿Qué es MVVM? Proporcione un ejemplo concreto de uso. ¿Qué otras arquitecturas existen y cuáles son sus diferencias principales?

MVVM (Model-View-ViewModel) es un patrón arquitectónico que separa la UI de la lógica en tres capas:

  • Model: datos y reglas de negocio (entidades, repositorios).
  • View: la UI (Composables). Solo describe qué mostrar.
  • ViewModel: mantiene el estado de UI, procesa eventos, coordina con el Model. Sobrevive recomposiciones y rotaciones.

La diferencia clave: el ViewModel no conoce la View; expone estado observable (StateFlow) y la View se suscribe. Esto produce un Unidirectional Data Flow puro.

Ejemplo concreto (login):

class LoginViewModel(repo: UsuarioRepository) : ViewModel() {
    private val _estado = MutableStateFlow(LoginUiState())
    val estado: StateFlow<LoginUiState> = _estado.asStateFlow()

    fun ingresar() { /* ... */ }
}

@Composable
fun LoginScreen(vm: LoginViewModel) {
    val estado by vm.estado.collectAsState()
    Button(onClick = vm::ingresar) { Text("Ingresar") }
}

Otras arquitecturas:

  • MVC (Model-View-Controller): el clásico. El Controller modifica el Model y la View se actualiza. En Android pre-Compose las Activities mezclaban todo, era difícil de mantener.
  • MVP (Model-View-Presenter): la View es pasiva, el Presenter le indica qué mostrar (vía interface). El Presenter conoce a la View, MVVM no.
  • Clean Architecture: complementaria a MVVM. Organiza toda la app en capas (UI → ViewModel → UseCase → Repository → DataSource) con regla de dependencia.
  • MVI (Model-View-Intent): estado inmutable + intents que producen un nuevo estado. Más estricto que MVVM.

En este curso se usa MVVM + Clean Architecture combinados.

Punto 4. ¿Qué es un StateFlow? ¿En qué se diferencia de un Flow normal y de LiveData? ¿Por qué es ideal para usarse en un ViewModel? Proporcione un ejemplo.

StateFlow es un Flow observable que siempre tiene un valor actual. Pertenece a las coroutines de Kotlin. Características clave:

  • Hot: el flujo está activo aunque nadie observe.
  • Tiene valor inicial obligatorio: siempre se puede pedir .value.
  • Conflado: solo emite si el valor realmente cambia.
  • Thread-safe: lectura y escritura concurrentes seguras.

Diferencias:

  • Un Flow normal es cold (se ejecuta cuando alguien lo collecta) y no tiene valor actual; sirve para streams de eventos. StateFlow es hot y representa estado.
  • LiveData es la versión legacy lifecycle-aware basada en observers, anterior a las coroutines. StateFlow es la opción moderna integrada con Kotlin.

Por qué es ideal en ViewModel: el ViewModel mantiene un MutableStateFlow privado y expone un StateFlow read-only. Compose lo observa con collectAsState() y se recompone al cambiar.

Ejemplo:

class UsuariosViewModel : ViewModel() {
    private val _estado = MutableStateFlow<UiState>(UiState.Loading)
    val estado: StateFlow<UiState> = _estado.asStateFlow()

    fun cargar() = viewModelScope.launch {
        _estado.value = UiState.Success(repo.obtener())
    }
}

  • DAO (Data Access Object): interface anotada con @Dao que define las operaciones (insert, update, delete, query).
  • Database: clase abstracta anotada con @Database que extiende RoomDatabase y agrupa entities + DAOs.

Ejemplo de uso (lista de usuarios):

    val email: String

    @Insert

Punto 2. Explique el patrón Repository y el patrón DataSource. ¿Qué problema resuelven? ¿Cuál es la diferencia entre ambos? Proporcione un ejemplo. Adicionalmente, explique Single Source of Truth y Offline First.

Repository: abstrae el origen de los datos. La UI y el ViewModel no deberían saber si los datos vienen de Room, Retrofit, DataStore, memoria, archivos o backend. Permite cambiar implementaciones, agregar cache o crear repositorios falsos para tests sin tocar la UI.

DataSource: representa una fuente concreta de datos (Room → LocalDataSource, Retrofit → RemoteDataSource, DataStore → PreferencesDataSource).

Diferencia: el DataSource sabe cómo hablar con una fuente concreta. El Repository decide cómo combinar fuentes, resolver conflictos, mapear datos y exponer una API coherente al resto de la app.

Ejemplo:

interface UsuarioRepository {
    suspend fun obtenerUsuarios(): List<Usuario>
}

class UsuarioRepositoryImpl(
    private val local: UsuarioLocalDataSource,
    private val remote: UsuarioRemoteDataSource
) : UsuarioRepository {
    override suspend fun obtenerUsuarios() = local.obtener()
}

Single Source of Truth: una parte del sistema es la fuente autorizada de un dato. Tener copias múltiples e independientes (en UI, ViewModel, Room, API) produce inconsistencias imposibles de debuggear. Lo correcto es: API → Repository → Room → ViewModel → UI, donde Room es la fuente de verdad observable.

Offline First: estrategia donde la app está diseñada para funcionar sin conexión. La UI lee siempre desde la base local (Room) y el Repository sincroniza con la API en background. La sincronización es responsabilidad explícita del Repository, no de la UI. Resultado: el usuario puede usar la app sin red.

Punto 3. ¿Qué es un Composable? ¿Qué es la recomposición? Considere una pantalla con un campo de texto y un texto que muestra el saludo. Enumere y explique los conceptos clave (remember, mutableStateOf, state hoisting) que intervienen.

Composable: función Kotlin marcada con @Composable que describe una porción de UI a partir del estado. No devuelve valor; solo se llama desde otro composable o desde setContent. Es la unidad básica de Jetpack Compose.

Recomposición: proceso por el cual Compose vuelve a ejecutar funciones composables cuando cambian sus parámetros o el estado que observan. Es selectiva: solo se re-ejecutan los composables afectados, no toda la pantalla.

Ejemplo de pantalla con campo + saludo:

@Composable
fun PantallaSaludo() {
    var nombre by remember { mutableStateOf("") }
    Column {
        TextField(
            value = nombre,
            onValueChange = { nombre = it }
        )
        Text("Hola, $nombre")
    }
}

Conceptos clave:

  • mutableStateOf(""): crea un estado observable. Cuando se modifica, Compose se entera y dispara la recomposición.
  • remember { ... }: conserva el valor entre recomposiciones. Sin remember, cada recomposición reiniciaría el estado.
  • by: delegación de propiedad. Permite escribir nombre en lugar de nombre.value.
  • State hoisting: si quisiera reutilizar este composable, lo separaría así: 
    @Composable
fun PantallaSaludo() {
    var nombre by remember { mutableStateOf("") }
    SaludoContent(nombre, onNombreChange = { nombre = it })
}

@Composable
fun SaludoContent(nombre: String, onNombreChange: (String) -> Unit) {
    /* solo describe UI, no guarda estado */
}
    El composable hijo es ahora stateless: recibe valor y callback. Es reutilizable, testeable y previewable.
Punto 4. Explique los patrones de diseño Builder, Proxy y Adapter. Para cada uno: problema que resuelve, idea del patrón, y un ejemplo concreto en Android.

Los tres son patrones GoF (Gang of Four, 1994) muy presentes en Android.

Builder:

  • Problema: construir objetos con muchos parámetros opcionales sin caer en constructores con 15 argumentos.
  • Idea: un objeto auxiliar recibe la configuración paso a paso y al final construye el objeto real.
  • Ejemplo Android: NotificationCompat.Builder(...).setContentTitle(...).setSmallIcon(...).build(). También AlertDialog.Builder, Retrofit.Builder, Room.databaseBuilder.

Proxy:

  • Problema: controlar el acceso a un objeto (lazy init, caching, llamadas remotas, logging) sin que el cliente sepa que hay un intermediario.
  • Idea: un objeto con la misma interfaz que el real, pero que intercepta las llamadas.
  • Ejemplo Android: Retrofit genera proxies dinámicos de tus interfaces de API. Cuando llamás api.getUsuarios(), parece una llamada local pero el proxy hace una request HTTP. Binder IPC también funciona como proxy.

Adapter:

  • Problema: hacer que dos interfaces incompatibles trabajen juntas.
  • Idea: convertir la interfaz de una clase en otra que el cliente espera.
  • Ejemplo Android: RecyclerView.Adapter. RecyclerView espera una interfaz fija (getItemCount, onCreateViewHolder, onBindViewHolder). El Adapter conecta tu lista de datos con esa interfaz. También los mappers entre DTO ↔ Entity ↔ Domain son adapters conceptuales.

Diferencia en una línea: Builder construye, Proxy representa, Adapter traduce.

Punto 1. Explique la relación entre Java y Kotlin en el contexto del desarrollo Android. ¿Por qué Google adoptó Kotlin? ¿Pueden coexistir en un mismo proyecto? Justifique técnicamente.

Kotlin es un lenguaje desarrollado por JetBrains que compila al mismo bytecode JVM que Java (archivos .class). Esa es la base técnica de su relación: ambos lenguajes producen el mismo formato intermedio, que en Android es procesado por D8 a .dex y ejecutado por ART.

  • Null safety en el sistema de tipos: String? vs String evita NullPointerExceptions en compile-time.
  • Coroutines nativas: la base del código asíncrono moderno (suspend, Flow, StateFlow). En Java son mucho más verbose.
  • Más conciso: data classes, properties, smart casts, expresiones en lugar de statements.
  • Funciones de extensión: agregar métodos sin heredar.
  • Lambdas y funciones de orden superior sin la verbosidad de Java.

.java (javac) ─┐
               ├─→ .class → D8 → .dex → ART
.kt   (kotlinc)─┘
Punto 2. Explique el modelo de runtime de Android: qué es ART, qué son JIT y AOT, y cómo se combinan en Android moderno. Mencione las ventajas y desventajas de cada estrategia.

ART (Android Runtime) es el entorno de ejecución por defecto desde Android 5.0. Reemplazó a Dalvik. Su entrada es bytecode .dex (Dalvik Executable), que es el formato Android obtenido al pasar bytecode JVM por D8.

JIT (Just-In-Time): compila en runtime, solo los métodos que se están usando. El bytecode se interpreta y los métodos "calientes" se compilan a código nativo y se cachean.

  • + Arranque rápido (no hay que compilar todo antes). Adaptativo al uso real. Menos espacio en disco.
  • − Pausas durante la compilación en runtime. Mayor uso de RAM para cache. Latencia inicial hasta "entrar en calor".

AOT (Ahead-Of-Time): compila antes de ejecutar, típicamente en la instalación. El bytecode se traduce a binario nativo guardado en disco.

  • + Arranque inmediato. Performance estable. Menor consumo de batería.
  • − Más espacio en disco. Instalación lenta. Menos adaptativo.

El modelo híbrido moderno (Android 7+): ART combina ambos progresivamente:

  1. Instalación rápida sin compilación pesada (solo verificación + quickening del .dex).
  2. Primera ejecución con JIT: la app arranca rápido y el runtime registra qué métodos son "hot" en un perfil.
  3. Compilación AOT en background (dex2oat) cuando el dispositivo está cargando e idle, guiada por el perfil. Genera archivos .oat (nativo) y .vdex (DEX verificado).
  4. Profile-on-Cloud: Google distribuye perfiles de uso típico junto con las apps, así el AOT se aplica desde el momento cero, antes incluso de que el usuario las use.

Resultado: instalaciones rápidas + arranques cada vez más veloces + menor uso de almacenamiento que un AOT total.

Punto 3. ¿Qué es el patrón Observer? ¿Cómo se manifiesta en Android moderno? Mencione al menos tres mecanismos donde aparece y dé un ejemplo de uso con StateFlow.

Observer es uno de los patrones GoF clásicos. Un objeto (el observable o subject) notifica automáticamente a múltiples objetos suscriptos (observers) cuando su estado cambia. Es la base del modelo reactivo.

Cómo se manifiesta en Android moderno:

  • StateFlow / Flow: la versión moderna basada en coroutines. collectAsState() en Compose o collect { } en una corrutina actúan como observers.
  • LiveData: histórico, lifecycle-aware. liveData.observe(this) { ... }.
  • Compose: la propia recomposición es Observer a gran escala. Compose registra qué estado lee cada composable y lo re-ejecuta cuando cambia.
  • RxJava: librería completa basada en Observer (Observable.subscribe(observer)), pre-coroutines pero todavía vigente en código legacy.

// Observable: el ViewModel expone estado

    private val _estado = MutableStateFlow<UiState>(UiState.Loading)
    val estado: StateFlow<UiState> = _estado.asStateFlow()
}

// Observer: el composable observa con collectAsState
@Composable
fun Pantalla(vm: UsuariosViewModel) {
    val estado by vm.estado.collectAsState()  // suscripción
    // cuando _estado.value cambia en el VM, esto recompone
    when (estado) {
        is UiState.Loading -> CircularProgressIndicator()
        is UiState.Success -> Lista(estado.usuarios)
    }
}

Componentes que intervienen:

  • UI (Composable): observa el StateFlow del ViewModel y se recompone.
  • ViewModel: pide al Repository observar usuarios; expone un StateFlow.
  • Repository: combina LocalDataSource (Room) y RemoteDataSource (Retrofit). Aplica Offline First: la UI observa la base local; el Repository sincroniza desde la API en background.
  • LocalDataSource: el DAO de Room que devuelve un Flow<List<UsuarioEntity>>.
  • RemoteDataSource: la interface de Retrofit que devuelve List<UsuarioDto>.
  • Mappers: convierten DTO ↔ Entity ↔ Domain Model.

Implementación conceptual:

class UsuarioOfflineRepository(
    private val dao: UsuarioDao,
    private val api: UsuarioApi
) : UsuarioRepository {

    override fun observarUsuarios(): Flow<List<Usuario>> =
        dao.observarUsuarios().map { it.map { e -> e.toDomain() } }

    suspend fun sincronizar() {
        val remotos = api.obtenerUsuarios()
        dao.reemplazarTodos(remotos.map { it.toEntity() })
    }
}

Beneficios: la app funciona sin conexión (Single Source of Truth = Room), la sincronización es responsabilidad explícita del Repository, y el ViewModel/UI no necesitan saber nada de la red.

Punto 5. Considerando una app que debe consumir un servicio API REST y mostrar los datos al usuario, explique qué es el asincronismo y por qué se utiliza para consumir APIs. Mencione las herramientas que provee Kotlin/Android para manejar asincronismo. Proporcione un ejemplo simple usando coroutines y un ViewModel.

Asincronismo es la capacidad de iniciar una operación que tarda (red, disco, cálculo pesado) sin bloquear el hilo principal. La operación se ejecuta en otro hilo o de forma cooperativa, y cuando termina notifica el resultado.

Por qué se usa para consumir APIs: el hilo principal de Android es el UI Thread. Si lo bloqueás haciendo una llamada HTTP que tarda 2 segundos, la app se congela y el sistema puede mostrar un ANR (Application Not Responding). Las llamadas a APIs siempre deben ejecutarse fuera del hilo principal.

Herramientas que provee Kotlin/Android:

  • Coroutines (la opción moderna): funciones suspend que se ejecutan de forma cooperativa sin bloquear el hilo.
  • Flow / StateFlow: streams asíncronos.
  • viewModelScope: scope de coroutines que vive con el ViewModel y se cancela automáticamente al destruirlo.
  • Retrofit + suspend functions: integración nativa con coroutines.
  • Legacy: AsyncTask (deprecado), Threads y Handlers, RxJava.

Ejemplo con coroutines y ViewModel:

// API con Retrofit
interface UsuariosApi {
    @GET("usuarios")
    suspend fun obtener(): List<UsuarioDto>
}

// ViewModel
class UsuariosViewModel(private val api: UsuariosApi) : ViewModel() {
    private val _estado = MutableStateFlow<UiState>(UiState.Loading)
    val estado: StateFlow<UiState> = _estado.asStateFlow()

    fun cargar() {
        viewModelScope.launch {                  // arranca corrutina
            _estado.value = UiState.Loading
            try {
                val data = api.obtener()         // suspend, no bloquea
                _estado.value = UiState.Success(data)
            } catch (e: Exception) {
                _estado.value = UiState.Error(e.message ?: "")
            }
        }
    }
}

// UI
@Composable
fun Pantalla(vm: UsuariosViewModel) {
    val estado by vm.estado.collectAsState()
    LaunchedEffect(Unit) { vm.cargar() }
    // render según estado
}

Punto 1. Explique el Stanford Design Thinking Process. Mencione y describa cada una de sus fases. ¿Por qué se considera un proceso no lineal? Proporcione un ejemplo concreto aplicado al desarrollo de una app móvil.

El Stanford Design Thinking Process (SDTP) es una metodología de innovación centrada en el ser humano, desarrollada por la d.school de Stanford. Provee una hoja de ruta para resolver problemas complejos y construir productos que satisfagan necesidades reales del usuario.

Por qué es no lineal: no es una secuencia rígida de pasos. Es un ciclo iterativo: cuando una fase posterior revela información nueva, el equipo vuelve a una fase anterior. Por ejemplo, si los tests muestran que el problema estaba mal definido, se vuelve a Empathize/Define.

Las 5 fases:

  1. Empathize: ponerse en el lugar del usuario. Entender necesidades, deseos, miedos, motivaciones. Herramientas: entrevistas, encuestas, shadowing, análisis de datos.
  2. Define: destilar los hallazgos en un problema claro y accionable. Pasar de "la app es lenta" a "los usuarios se frustran al subir fotos porque la carga es lenta en redes móviles".
  3. Ideate: brainstorming sin juzgar. Generar la mayor cantidad de soluciones posibles, incluso las "locas". Divergir y luego converger.
  4. Prototype: construir representaciones tangibles y de bajo costo (sketches, wireframes, simulaciones). Objetivo: validar rápido, no perfeccionar. "Fail fast, iterate quickly".
  5. Test: probar prototipos con usuarios reales. Identificar puntos de fricción. Los hallazgos pueden mandar al equipo de vuelta a fases anteriores.

Ejemplo aplicado · app de reservas de turnos médicos:

  • Empathize: entrevistar a 10 personas mayores sobre cómo agendan turnos. Descubrir que muchas se frustran con apps que tienen letra chica.
  • Define: "los usuarios mayores de 60 años abandonan el flujo de reserva porque no pueden leer cómodamente las opciones disponibles".
  • Ideate: brainstorm de soluciones: tipografía grande por default, modo voz, asistente paso a paso, integración con familiar.
  • Prototype: diseñar dos prototipos en Figma con tipografía grande y wizard simplificado.
  • Test: probar con 5 usuarios mayores. Detectar que el wizard funciona bien pero el modo voz confunde. Iterar: volver a Define, ajustar el flujo.
Punto 2. Explique la Ley de Fitts. Indique la fórmula y el significado de sus variables. ¿Qué implicancias tiene para el diseño de interfaces táctiles? Proporcione al menos tres recomendaciones concretas para una app móvil.

La Ley de Fitts es una ley empírica de la HCI que establece que el tiempo necesario para alcanzar un objetivo depende de la distancia hasta el objetivo y del tamaño del mismo.

Fórmula:

T = a + b · log2(D/W + 1)

Variables:

  • T = tiempo de movimiento.
  • D = distancia hasta el objetivo.
  • W = ancho del objetivo.
  • a, b = constantes empíricas que dependen del dispositivo (mouse, joystick, dedo).

Lo que dice la fórmula: el tiempo aumenta logarítmicamente con la distancia y disminuye logarítmicamente al aumentar el tamaño. Más grande y más cerca → más rápido alcanzar.

Implicancias para diseño táctil mobile:

  1. Objetivos grandes son más rápidos. Botones primarios (CTA) deben ser grandes. Material Design recomienda mínimo 48×48 dp.
  2. Objetivos cercanos son más rápidos. Las acciones frecuentes deben estar cerca del punto de atención (zona del pulgar en mobile).
  3. Bordes y esquinas son "infinitos". Un objetivo en el borde de la pantalla es muy rápido de alcanzar porque el dedo no puede pasarlo. Por eso el FAB va abajo a la derecha y el botón de back arriba a la izquierda.

Tres recomendaciones concretas para una app móvil:

  1. Touch targets de mínimo 48×48 dp incluso si visualmente son más chicos. Los íconos pueden verse de 24 dp pero el área tappable debe ser de 48 dp.
  2. Acciones primarias en la zona del pulgar (parte inferior de la pantalla). Los menús superiores son menos accesibles cuando se usa con una mano.
  3. Aprovechar bordes y esquinas para acciones críticas: FAB en esquina inferior derecha, navegación en bordes inferiores (BottomNavigation).
Punto 3. Explique la diferencia entre las arquitecturas RISC y CISC. ¿Qué arquitectura domina el mercado mobile actualmente y por qué? Realice una tabla comparativa entre ARM y x86 considerando filosofía, consumo de energía, ejecución y dominio del mercado.

El mundo de los microprocesadores está dominado por dos filosofías de diseño:

  • CISC (Complex Instruction Set Computing): pocas instrucciones por programa, pero cada una es compleja y ocupa varios ciclos de reloj. Diseñada para reducir el código que escribe el programador.
  • RISC (Reduced Instruction Set Computing): muchas instrucciones simples y uniformes; cada una se ejecuta típicamente en un solo ciclo. Surge en los 80s al notar que el rendimiento mejora con instrucciones simples y uniformes.

Arquitectura que domina el mercado mobile: ARM (Advanced RISC Machines), una arquitectura RISC. Cubre prácticamente el 100% de los smartphones actuales.

Por qué ARM ganó en mobile:

  • Bajo consumo de batería: instrucciones simples consumen menos energía. Crítico cuando la batería es el recurso más escaso.
  • Costo reducido de licencias: ARM Ltd. no fabrica chips, los licencia. Apple, Samsung, Qualcomm diseñan sus propios SoCs sobre ARM. Eso generó competencia y precios competitivos.
  • Flexibilidad para SoCs: ARM se integra fácilmente con GPU, modems, NPU, sensores en un único chip.

Frase de cierre: "x86 es un CISC que se volvió RISC por dentro, mientras ARM es RISC puro y eficiente por diseño". Hoy los chips Apple M-series demuestran que ARM puede superar a x86 incluso en laptops y desktops.

Punto 4. Explique qué es un SoC (System on Chip) y por qué es fundamental para los dispositivos móviles. Mencione los componentes que típicamente integra. ¿Qué relación tiene con la arquitectura ARM?

Un SoC (System on Chip) es un circuito integrado que contiene los componentes de una computadora completa en un único sustrato de silicio. Es la pieza que hace posible la electrónica mobile moderna.

Por qué es fundamental en mobile:

  • Tamaño reducido: los SoCs son del tamaño de una moneda; permiten integrar una computadora completa en el espacio mínimo de un teléfono.
  • Bajo consumo: optimizado para low-power, prolonga la duración de batería.
  • Flexibilidad: reprogramables y adaptables.
  • Confiabilidad: diseño relativamente sencillo, seguros.
  • Costo-eficiente: más barato que adquirir componentes separados.

Componentes que típicamente integra un SoC moderno:

  • CPU (cores ARM): lógica general.
  • GPU: gráficos, juegos, renderizado de UI.
  • DSP (Digital Signal Processor): audio, cámara, multimedia.
  • NPU / AI Engine: aceleración de Machine Learning on-device.
  • ISP (Image Signal Processor): procesamiento de fotografía y video.
  • Modem 4G/5G: conectividad móvil.
  • Sensores y controladores de energía.
  • Módulo de seguridad (ej: TrustZone).

Relación con ARM: ARM es la arquitectura base sobre la cual se construyen los SoCs mobile. ARM Ltd. licencia el set de instrucciones; los fabricantes (Apple, Qualcomm Snapdragon, Samsung Exynos, Huawei Kirin) diseñan SoCs completos integrando CPU ARM + el resto de los componentes.

Ejemplo concreto: el Qualcomm Snapdragon integra cores ARM, GPU Adreno, NPU Hexagon, modem 5G, ISP — todo en un único chip que cabe en un smartphone. Esto convierte al teléfono en computadora, cámara, consola y centro de IA al mismo tiempo.

Punto 5. Explique qué áreas intervienen en la Interacción Humano-Computadora (HCI). ¿Qué relación existe entre HCI, UX e UI? Mencione al menos tres heurísticas de Nielsen aplicables a mobile. Realice un wireframe de una pantalla aplicando dichas heurísticas Y considerando la Ley de Fitts.

La HCI (Human-Computer Interaction) es el campo multidisciplinario que estudia cómo las personas interactúan con sistemas computacionales y cómo diseñarlos para que sean efectivos, eficientes y satisfactorios.

Áreas que intervienen: Computer Science / Engineering (capa técnica), Psicología cognitiva (cómo procesamos información), Ergonomía y human factors (adaptación física), Diseño gráfico/industrial, Lingüística (lenguaje, copywriting), Sociología/antropología/etnografía (contexto cultural), Semiótica y branding (significado de signos).

Relación HCI / UX / UI:

  • UI ⊂ UX ⊂ HCI.
  • UI es la capa visual concreta (color, tipografía, layout).
  • UX es la práctica profesional centrada en el usuario: usabilidad, flujos, lógica de interacción. Contiene UI.
  • HCI es el marco académico multidisciplinario que estudia toda la interacción humano-máquina. Contiene UX.

Tres heurísticas de Nielsen aplicables a mobile:

  1. Visibilidad del estado del sistema: el usuario debe saber siempre qué está pasando. En mobile, donde la red es variable, mostrar progreso es crítico (loaders, indicadores de subida, snackbars de confirmación).

Punto 6. Desarrolle el concepto de diseño centrado en el contexto en aplicaciones móviles. ¿Qué aspectos del contexto debe tener en cuenta un diseñador? Describa un caso hipotético de aplicación que aproveche el contexto, destacando la UX y la accesibilidad.

El diseño centrado en el contexto es el enfoque por el cual una app móvil adapta su comportamiento a la situación real del usuario, en lugar de presentar siempre la misma experiencia genérica.

Por qué importa en mobile específicamente: a diferencia del PC, el contexto del usuario mobile cambia constantemente. La misma persona puede estar en casa con Wi-Fi, en la calle con 4G, en el subte sin señal, todo en pocos minutos. Una app que ignora el contexto da una experiencia genérica y frustrante.

Aspectos del contexto a considerar:

  • Condiciones ambientales: iluminación (sol directo, oscuridad), ruido, distracciones.
  • Tareas y objetivos: por qué el usuario abrió la app en este momento.
  • Conexión de red: calidad y velocidad disponibles.
  • Disponibilidad de datos: Wi-Fi vs plan limitado.
  • Interacción física: uso con una sola mano, gestos, voz.
  • Privacidad: datos sensibles en pantallas públicas.
  • Sensores disponibles: GPS, acelerómetro, cámara, NFC.
  • Hora del día y batería disponible.

Caso hipotético detallado · app de transporte público:

Contexto detectado: GPS (ubicación), hora (rush hour o no), conectividad (Wi-Fi, 4G, sin señal), batería actual, sensor de luz ambiente.

Adaptaciones del diseño:

  • Al abrir la app, mostrar la parada más cercana usando GPS, sin pedir buscarla.
  • En rush hour, mostrar tiempos de espera más prominentes y alertas de demoras.
  • Con 4G y datos limitados, cargar versión liviana del mapa.
  • Detectar pérdida de señal (entrada al subte): activar modo offline con horarios cacheados.
  • Si la batería está baja, reducir actualizaciones GPS a cada 60 segundos.
  • Con sol directo (sensor de luz), aumentar contraste / activar modo claro.

UX resultante: reducción de esfuerzo cognitivo (el usuario no configura nada), confiabilidad (funciona aunque caiga la red), personalización percibida ("la app me entiende"), eficiencia (un toque para info crítica).

Accesibilidad incorporada:

  • Touch targets de 48 dp mínimos: importante para personas con problemas motores.
  • Contraste alto para baja visión y sol directo.
  • Tipografía escalable que respete el tamaño configurado por el usuario.
  • Soporte de lectores de pantalla (TalkBack) con contentDescription.
  • No depender solo del color: demoras se indican con color rojo + ícono + texto.
  • Animaciones reducibles respetando "reducir movimiento" del sistema.
